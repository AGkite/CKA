## 一、网络模型总览

Kubernetes 网络有几条基本规则：

- **每个 Pod 有独立 IP**，Pod 间可直接用 IP 互通（由 CNI 如 Calico 实现）。
- **Pod IP 会变**（重建就换），所以不能依赖 Pod IP，要用 **Service**。
- **Service 提供稳定入口**，把流量负载均衡到一组 Pod。
- **CoreDNS 提供域名解析**，把 Service 名解析成 ClusterIP。

```
外部用户
   │ NodePort / Ingress
   ↓
Service（稳定 IP/名字）──selector──> 一组 Pod（IP 会变）
   ↑ 解析名字
CoreDNS
```

三层职责（第一部分问答里出现过）：

| 组件 | 职责 |
|------|------|
| **Calico（CNI）** | 给 Pod 分 IP、打通跨节点网络（「修路」） |
| **kube-proxy** | 把 Service 的访问转发到后端 Pod（「导航规则」） |
| **CoreDNS** | 域名 → ClusterIP（「电话簿」） |

本机是双网卡场景时，节点可能同时有 WSL/NAT 管理网卡和固定集群网卡。本文所有节点间访问与 NodePort 验证均使用 `eth1` 的固定地址 `10.13.31.10/11/12`，避免节点地址随 NAT 变化。Calico 使用 IPIP 时，跨节点 Pod 流量会封装后经 `eth1` 传输；Pod 地址仍来自 `192.168.0.0/16`，不要把 Pod CIDR 与节点网段混为一谈。

## 二、Service 四种类型

| 类型 | 说明 | 场景 |
|------|------|------|
| **ClusterIP** | 分配一个**集群内部**虚拟 IP（默认） | 集群内服务互访 |
| **NodePort** | 在每个节点开一个端口（30000–32767） | 集群外访问、无 LB 环境 |
| **LoadBalancer** | 云厂商负载均衡器 + NodePort | 生产对外（本地环境通常无 LB） |
| **ExternalName** | 返回一个 CNAME 指向外部域名 | 代理集群外服务 |

### 2.1 创建 Service

不要对同名 `web` 连续执行两次 `kubectl expose`，第二次会因 Service 已存在而失败。下面用不同名字创建一个内部 Service；现有对外 Service `dev/web` 在下一段用声明式 YAML统一为固定 NodePort。

**执行位置：控制端 WSL Bash；操作目标：为 `dev/web` Deployment 创建 `dev/web-internal` ClusterIP Service。**

```bash
kubectl expose deployment web --name=web-internal \
  --port=80 --target-port=80 -n dev
```

| 参数 | 含义 |
|------|------|
| `expose deployment web` | 为 web 下的 Pod 建 Service（按 label 关联） |
| `--name=web-internal` | 显式使用不同的 Service 名，避免与现有 `web` 重名 |
| `--port=80` | Service 自己的端口 |
| `--target-port=80` | 转发到 Pod 容器的端口 |
| `--type=NodePort` | 类型；不写默认 ClusterIP |

### 2.2 固定现有 Service 的 NodePort

为避免 `kubectl expose` 随机分配端口，使用完整清单声明固定 `nodePort: 31631`。先确认 `dev/web` Deployment 的 Pod 标签包含 `app: web`；若实际标签不同，应同步修改 selector。

**完整清单：在控制端保存为 `web-service.yaml`，再从控制端 `apply`；会创建或更新现有 `dev/web` Service，不会重复创建同名资源。**

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web
  namespace: dev
spec:
  type: NodePort
  selector:
    app: web
  ports:
  - name: http
    protocol: TCP
    port: 80
    targetPort: 80
    nodePort: 31631
```

**执行位置：控制端 WSL Bash；操作目标：创建或更新 `dev/web` Service。**

```bash
kubectl apply -f web-service.yaml
kubectl get svc web -n dev
```

### 2.3 从外部访问 NodePort

**执行位置：控制端 WSL Bash；操作目标：先查看 `dev/web`，再分别访问三台节点 `eth1` 固定 IP 的同一 NodePort。**

```bash
kubectl get svc web -n dev
# PORT(S): 80:31631/TCP  →  31631 是 NodePort

curl http://10.13.31.10:31631      # k8s-master eth1
curl http://10.13.31.11:31631      # k8s-worker1 eth1
curl http://10.13.31.12:31631      # k8s-worker2 eth1
```

> **注意：** NodePort 通常可通过任一节点 IP 访问，即使后端 Pod 不在该节点；若某一个 `eth1` IP 不通，检查节点防火墙、kube-proxy 与路由。不要 `ping` ClusterIP（它是虚拟 IP，不响应 ICMP），访问服务用 TCP（curl/wget）。

## 三、端口三兄弟：port / targetPort / nodePort

这三个最容易混，务必分清：

**以下是 Service `spec` 片段，不能单独 `apply`；完整可应用清单见 2.2。**

```yaml
spec:
  type: NodePort
  selector:
    app: web
  ports:
  - port: 80          # ① Service 端口：集群内访问 Service 用
    targetPort: 80    # ② Pod 容器端口：流量最终送到这里
    nodePort: 31631   # ③ 节点端口：集群外访问用（NodePort 才有）
```

| 端口 | 谁用 | 说明 |
|------|------|------|
| `port` | 集群内 | `curl web.dev.svc:80` |
| `targetPort` | Pod | 必须等于容器实际监听端口 |
| `nodePort` | 集群外 | `curl 节点IP:31631`，范围 30000–32767 |

流量路径：

```
集群外 → 节点IP:nodePort(31631) → Service:port(80) → Pod:targetPort(80)
```

> **高频错误：** `targetPort` 与容器实际端口不一致，导致 Service 通但返回不了内容。

## 四、Endpoints 与 selector 匹配

### 4.1 Endpoints 是什么

**Service** 是稳定入口，**Endpoints** 是它背后**当前**真实的 Pod IP:端口列表，由控制器根据 selector 自动维护。

**执行位置：控制端 WSL Bash；操作目标：`dev/web` Service 的后端。**

```bash
kubectl get endpoints web -n dev
# web   192.168.126.2:80,192.168.194.66:80,...
```

### 4.2 排错核心：Endpoints 为空

**执行位置：控制端 WSL Bash；操作目标：`dev/web` Service 的后端。**

```bash
kubectl get endpoints web -n dev
# 若显示 <none> 或为空 → Service 选不到 Pod
```

原因几乎都是 **Service 的 selector 与 Pod 的 label 不匹配**：

**执行位置：控制端 WSL Bash；操作目标：对比 `dev/web` Service selector 与 `dev` Pod 标签。**

```bash
kubectl get svc web -n dev -o yaml | grep -A3 selector    # 看 Service 要什么 label
kubectl get pods -n dev --show-labels                     # 看 Pod 实际 label
```

只要 Pod 带的 label 包含 Service selector 要求的键值，就会进 Endpoints。

> **考试高频：** 「Service 访问不通」十有八九是 selector 拼错，或 targetPort 错。先查 Endpoints。

### 4.3 EndpointSlices（新版机制，了解）

新版 Kubernetes 用 **EndpointSlice** 替代传统 Endpoints（把大列表切成小片，更高效）。两者信息类似，排查时都可看：

**执行位置：控制端 WSL Bash；操作目标：题目指定命名空间中的 Service/EndpointSlice（替换占位符）。**

```bash
kubectl get endpoints <svc> -n <ns>        # 传统视图（仍可用）
kubectl get endpointslices -n <ns>         # 新机制，按 Service 分片
kubectl get endpointslices -n <ns> -l kubernetes.io/service-name=<svc>
```

> 考试若题目提到 EndpointSlice，用 `kubectl get endpointslices` 查看；排错逻辑与 Endpoints 一致（后端为空 = selector/label 不匹配）。

## 五、集群 DNS 与 CoreDNS

### 5.1 DNS 命名规则

| 对象 | 完整域名（FQDN） |
|------|------------------|
| Service | `<service>.<namespace>.svc.cluster.local` |
| 同命名空间短名 | `<service>` |
| 跨命名空间 | `<service>.<namespace>` |
| Pod | `<pod-ip-用横线>.<namespace>.pod.cluster.local` |

例：`dev` 里的 Pod 访问同命名空间的 `web`，直接 `curl http://web` 即可；访问 `prod` 里的 `db`，用 `curl http://db.prod`。

### 5.2 CoreDNS

- 集群 DNS 由 **CoreDNS** 实现（两个 Pod，在 `kube-system`）。
- 对外的 Service 名叫 **`kube-dns`**（兼容旧名），ClusterIP 通常是 `10.96.0.10`。
- 每个 Pod 的 `/etc/resolv.conf` 里 `nameserver` 指向它。

**执行位置：控制端 WSL Bash；操作目标：`kube-system` 中的 CoreDNS 资源。**

```bash
kubectl get pods -n kube-system -l k8s-app=kube-dns       # CoreDNS Pod
kubectl get svc kube-dns -n kube-system                   # DNS Service
kubectl get configmap coredns -n kube-system -o yaml      # CoreDNS 配置
```

### 5.3 DNS 排查

**执行位置：控制端 WSL Bash；操作目标：临时测试 Pod及 `dev` 中指定 Pod（替换 `<pod>`）。**

```bash
# 起一个临时 Pod 测试解析
kubectl run test-dns --image=busybox:1.36 --rm -it -- nslookup kubernetes.default

# 看某 Pod 的 DNS 配置
kubectl exec <pod> -n dev -- cat /etc/resolv.conf
```

| 命令片段 | 含义 |
|----------|------|
| `--rm` | 退出即删除临时 Pod |
| `-it` | 交互式终端 |
| `nslookup kubernetes.default` | 解析默认命名空间的 kubernetes Service |

`resolv.conf` 里的 `search` 和 `ndots:5` 决定短名如何自动补全后缀。

## 六、Ingress

### 6.1 概念

**Service（NodePort）** 每个应用要占一个端口，难管理。**Ingress** 在 Service 前再加一层 **七层（HTTP/HTTPS）入口**，用**域名 + 路径**把请求分发到不同 Service。

```
用户 → web.example.com/api → Ingress → api-service → Pod
      web.example.com/     → Ingress → web-service → Pod
```

> **关键：** Ingress 资源本身**不转发流量**，必须先装 **Ingress Controller**（如 ingress-nginx）才生效。
>
> **本教程不假设当前集群已安装 Ingress Controller。** 应先确认存在可用控制器和 IngressClass，并确保后端 `dev/api-service`、可选 TLS Secret `dev/tls-secret` 已创建；否则下面资源即使被 API Server 接受，也不会产生可用入口。

### 6.2 Ingress YAML

**完整清单：仅当前置的 Ingress Controller、`nginx` IngressClass、`dev/api-service`（以及启用 TLS 时的 `dev/tls-secret`）真实存在时，才在控制端保存并 `apply`。示例不表示这些资源已安装。**

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  namespace: dev
spec:
  ingressClassName: nginx        # ← 指定用哪个 Ingress Controller（v1.22+）
  rules:
  - host: myapp.example.com      # 按域名匹配
    http:
      paths:
      - path: /api
        pathType: Prefix         # Exact | Prefix | ImplementationSpecific
        backend:
          service:
            name: api-service    # 转发到哪个 Service
            port:
              number: 80
  tls:                           # 可选：HTTPS
  - hosts:
    - myapp.example.com
    secretName: tls-secret       # 存证书的 Secret
```

| 字段 | 含义 |
|------|------|
| `ingressClassName` | 指定 Ingress Controller，必填 |
| `host` | 匹配的域名 |
| `path` + `pathType` | 匹配的路径；`Prefix` 最常用 |
| `backend.service` | 转发到的 Service 和端口 |
| `tls` | 配 HTTPS，引用证书 Secret |

### 6.3 命令式创建（快速）

**执行位置：控制端 WSL Bash；操作目标：创建 `dev/my-ingress`；执行前置条件同 6.2。**

```bash
kubectl create ingress my-ingress \
  --rule="myapp.example.com/api=api-service:80" \
  --class=nginx -n dev
```

## 七、Gateway API

Gateway API 是 Ingress 的**下一代**，把「基础设施」和「路由」职责分开，功能更强。考试主要要求**理解概念、能照文档改**。

> **本教程不假设当前集群已安装 Gateway API。** 使用前必须同时具备 Gateway API CRD、对应实现/控制器、一个真实存在且 `Accepted=True` 的 GatewayClass，以及后端 `dev/api-service`。`gatewayClassName: nginx` 只是示例名，不能据此推断集群里已存在该类。

| 资源 | 角色 |
|------|------|
| **GatewayClass** | 定义用哪种实现（类似 ingressClass） |
| **Gateway** | 基础设施层：监听哪些端口/协议 |
| **HTTPRoute** | 应用层：域名、路径 → 后端 Service |
| **TCPRoute / GRPCRoute / UDPRoute** | 其他协议路由（部分位于 Experimental CRD，取决于安装版本） |

**完整清单：仅满足上述 CRD、控制器、GatewayClass 和后端 Service 前置条件时，才在控制端保存并 `apply`；示例不表示这些组件已安装。**

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: my-gateway
  namespace: dev
spec:
  gatewayClassName: nginx
  listeners:
  - name: http
    protocol: HTTP
    port: 80
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-route
  namespace: dev
spec:
  parentRefs:
  - name: my-gateway          # 挂到哪个 Gateway
  hostnames:
  - "myapp.example.com"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /api
    backendRefs:
    - name: api-service
      port: 80
```

对照记忆：**Gateway ≈ Ingress 的入口部分；HTTPRoute ≈ Ingress 的 rules 部分。**

### 7.1 Ingress → Gateway API 迁移（2026 新题型）

考纲明确要求「用 Gateway API 管理 Ingress 流量」，可能考**把已有 Ingress 迁移成 Gateway + HTTPRoute**。对应关系：

| Ingress 里的 | 迁移到 Gateway API |
|--------------|--------------------|
| `ingressClassName: nginx` | `Gateway.spec.gatewayClassName` |
| 监听 80/443 | `Gateway.spec.listeners`（port/protocol） |
| `tls.secretName` | `Gateway` listener 的 `tls.certificateRefs` |
| `rules[].host` | `HTTPRoute.spec.hostnames` |
| `paths[].path` + `backend.service` | `HTTPRoute.spec.rules[].matches` + `backendRefs` |

迁移步骤：① 建 `Gateway`（定义监听端口 + TLS）→ ② 建 `HTTPRoute`（域名/路径 → 后端 Service，`parentRefs` 指向 Gateway）→ ③ 验证后删除旧 Ingress。

> TLS 证书仍用 Secret，只是在 Gateway 的 listener 里用 `certificateRefs` 引用。

## 八、NetworkPolicy 网络策略

### 8.1 概念

默认情况下，**所有 Pod 之间都能互通**。**NetworkPolicy** 用来限制「谁能访问谁」，相当于 Pod 级防火墙。

**关键规则：**

- 一旦某 Pod 被某个 NetworkPolicy 的 `podSelector` 选中，**该方向（Ingress/Egress）默认全部拒绝**，只放行明确列出的。
- 需要已安装且健康、并启用策略能力的 CNI；当前基线为 Calico，支持 Kubernetes NetworkPolicy，但仍应先确认 Calico Pod 正常。
- **加了 Egress 规则，一定要放行 DNS（UDP 53）**，否则 Pod 连域名都解析不了。

### 8.2 完整示例

前置资源：`production` 命名空间必须存在；被保护的 Pod 需带 `app=api`，允许的调用方需在同一命名空间且带 `app=frontend`，数据库 Pod 需在同一命名空间且带 `app=database`。端口也必须与应用实际监听一致。本教程不声称这些示例资源已经存在。

**完整清单：确认上述命名空间、Pod 标签、端口和 Calico 状态后，在控制端保存为 `api-policy.yaml` 并执行 `kubectl apply -f api-policy.yaml`。**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-policy
  namespace: production
spec:
  podSelector:                    # 这条策略作用在哪些 Pod 上
    matchLabels:
      app: api
  policyTypes:
  - Ingress                       # 管入站
  - Egress                        # 管出站
  ingress:
  - from:
    - podSelector:                # 只允许带 app=frontend 的 Pod 访问
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - podSelector:                # 只允许访问 app=database 的 Pod
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 5432
  - ports:                        # ★ 必须放行 DNS，否则解析失败
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
```

### 8.3 选择器 AND / OR（易错）

**以下是 NetworkPolicy `spec.ingress` 片段，不能单独 `apply`。**

```yaml
# AND：同一个 from 条目里，namespaceSelector 和 podSelector 同时满足
ingress:
- from:
  - namespaceSelector:
      matchLabels: { env: prod }
    podSelector:
      matchLabels: { app: frontend }

# OR：多个 from 条目，满足任意一个即可
ingress:
- from:
  - namespaceSelector:
      matchLabels: { env: prod }
- from:
  - podSelector:
      matchLabels: { app: frontend }
```

区别就在 `-`（列表项）的位置：同一项内是 AND，分成多项是 OR。

### 8.4 默认拒绝所有

**以下是 NetworkPolicy `spec` 片段，不能单独 `apply`；还需补齐 `apiVersion`、`kind` 和 `metadata.namespace`。**

```yaml
spec:
  podSelector: {}       # 选中命名空间内所有 Pod
  policyTypes:
  - Ingress
  - Egress
  # 不写 ingress/egress 规则 = 全部拒绝
```

## 九、CNI 插件

| 插件 | 支持 NetworkPolicy | 说明 |
|------|--------------------|------|
| **Calico** | ✅ | 考试最常用，你的集群用的就是它 |
| **Flannel** | ❌ | 简单，但不支持 NetworkPolicy |
| **Cilium** | ✅ | eBPF，功能强 |
| **Weave** | ✅ | — |

查看节点本地 CNI 配置时，必须在具体节点执行；先以 `k8s-master` 为例：

**执行位置：`k8s-master` 节点本地 Bash；可从 Windows PowerShell 执行 `multipass shell k8s-master` 进入节点；操作目标：master 本机 CNI 配置。**

```bash
ls /etc/cni/net.d/
```

Calico Pod 属于集群资源，回到控制端查询：

**执行位置：控制端 WSL Bash；操作目标：`kube-system` 中的 Calico Pod。**

```bash
kubectl get pods -n kube-system | grep calico
```

当前基线使用 Calico **IPIP**。双网卡节点上应确认 Kubernetes Node InternalIP 和 Calico 选中的节点地址来自固定 `eth1`（`10.13.31.0/24`），否则封装流量可能走到 WSL/NAT 网卡，引发跨节点 Pod 间歇性不通。

## 十、Pod 连通性排查

**执行位置：控制端 WSL Bash；操作目标：`dev` 工作负载及集群网络资源（替换尖括号占位符）。**

```bash
# Pod IP 与所在节点
kubectl get pods -o wide -n dev

# 从 A Pod 访问 B Pod 的 IP
kubectl exec <pod-a> -n dev -- wget -qO- --timeout=2 http://<pod-b-ip>:80

# 访问 Service（走 DNS）
kubectl exec <pod-a> -n dev -- curl http://web.dev.svc.cluster.local

# 完整网络排查链
kubectl get svc web -n dev              # ① Service 在不在
kubectl get endpoints web -n dev        # ② 有没有后端 Pod
kubectl get pods -n dev --show-labels   # ③ label 对不对
kubectl get networkpolicy -n dev        # ④ 有没有策略挡住
kubectl get pods -n kube-system -l k8s-app=kube-dns   # ⑤ DNS 正不正常
```

## 十二、易错点与自检清单

### 易错点

- ClusterIP **不能 ping**，用 curl/wget 测端口。
- `targetPort` 必须等于容器实际监听端口。
- Service 不通先查 **Endpoints**，多半是 selector / label 不匹配。
- NetworkPolicy 加 Egress 后**必须放行 DNS（UDP 53）**。
- Ingress 需要先有 **Ingress Controller** 才生效。
- Ingress 要写 `ingressClassName`。
- Gateway API 还需预装 CRD、实现控制器和真实存在的 GatewayClass。
- 双网卡集群要确认节点与 Calico 使用固定 `eth1` 地址，IPIP 的 Pod 网段不是节点网段。
