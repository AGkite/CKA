## 一、最小单元：Pod

### 1.1 什么是 Pod

**Pod** 是 Kubernetes 里**最小的调度和运行单元**。

- 一个 Pod 里可以有 **1 个或多个容器**，它们**共享同一个网络（同一个 IP）和存储卷**。
- 你不能单独调度「一个容器」，只能调度「一个 Pod」。
- 绝大多数情况一个 Pod 只放一个主容器；多容器用于「边车（sidecar）」模式（如日志收集）。

类比：Pod 像「一间公寓」，容器是「住在里面的人」，共享同一个门牌号（IP）和客厅（存储）。

### 1.2 命令式创建 Pod

```bash
kubectl run nginx --image=nginx:1.27 -n dev
```

| 部分 | 含义 |
|------|------|
| `kubectl run` | 创建单个 Pod（区别于 `create deployment`） |
| `nginx` | Pod 名字 |
| `--image=nginx:1.27` | 用哪个镜像 |
| `-n dev` | 放在 `dev` 命名空间（不写则是 `default`） |

### 1.3 生成 YAML（考试更常用）

```bash
kubectl run nginx --image=nginx:1.27 --dry-run=client -o yaml > pod.yaml
```

| 参数 | 含义 |
|------|------|
| `--dry-run=client` | **不真正创建**，只在本地模拟 |
| `-o yaml` | 以 YAML 格式输出 |
| `> pod.yaml` | 保存到文件，方便编辑 |

生成的 `pod.yaml`：

```yaml
apiVersion: v1          # 使用的 API 版本，Pod 属于核心组 v1
kind: Pod               # 资源类型
metadata:
  name: nginx           # 名字
  labels:
    run: nginx          # 标签（给 selector 用）
spec:                   # 期望状态
  containers:
  - name: nginx         # 容器名（滚动更新时会用到）
    image: nginx:1.27   # 镜像
    ports:
    - containerPort: 80 # 容器监听端口（仅声明作用）
```

改完后创建：

```bash
kubectl apply -f pod.yaml
```

### 1.4 查看与排查

```bash
kubectl get pod -n dev                 # 列表
kubectl get pod nginx -n dev -o wide   # 含 IP、所在节点
kubectl describe pod nginx -n dev      # 详情 + Events（排错核心）
kubectl logs nginx -n dev              # 容器日志
kubectl exec -it nginx -n dev -- sh    # 进容器
kubectl delete pod nginx -n dev        # 删除
```

| 命令 | 作用 |
|------|------|
| `describe` | 看事件（调度、拉镜像、启动失败原因） |
| `logs` | 看应用输出 |
| `logs --previous` | 看**上一次崩溃**的容器日志（CrashLoopBackOff 必用） |
| `exec -it ... -- sh` | 进容器交互调试 |

### 1.5 restartPolicy（重启策略）

```yaml
spec:
  restartPolicy: Always   # Always（默认）| OnFailure | Never
```

| 值 | 含义 | 常用于 |
|----|------|--------|
| `Always` | 容器退出总是重启 | Deployment 的 Pod |
| `OnFailure` | 非 0 退出才重启 | Job |
| `Never` | 从不重启 | 一次性任务 |

## 二、Deployment 与 ReplicaSet

### 2.1 为什么不用裸 Pod

裸 Pod 挂了不会自愈、不能扩缩容、不能滚动更新。生产用 **Deployment** 管理。

层级关系：

```
Deployment（你操作它）
   └── ReplicaSet（自动创建，维持副本数）
          └── Pod × N（真正运行）
```

| 对象 | 职责 |
|------|------|
| **Deployment** | 声明「我要 3 个 nginx」，管理版本、滚动更新、回滚 |
| **ReplicaSet** | 确保「实际数量 = 期望数量」，少了就补 |
| **Pod** | 实际运行容器 |

**你几乎只操作 Deployment，不直接碰 ReplicaSet。**

### 2.2 创建 Deployment

```bash
kubectl create deployment web --image=nginx:1.27 --replicas=3 -n dev
```

| 参数 | 含义 |
|------|------|
| `create deployment web` | 创建名为 web 的 Deployment |
| `--image=nginx:1.27` | 容器镜像 |
| `--replicas=3` | 维持 3 个 Pod |
| `-n dev` | 命名空间 |

查看三级资源：

```bash
kubectl get deploy,rs,pod -n dev
```

### 2.3 扩缩容

```bash
kubectl scale deployment web --replicas=5 -n dev
```

`--replicas=5`：把副本数调到 5，K8s 自动新建 2 个 Pod。缩容同理。

> 注意：`scale` 只改副本数，**不会**产生新的 revision（版本历史）。

### 2.4 Deployment YAML 结构（必须理解 selector）

```yaml
apiVersion: apps/v1        # Deployment 属于 apps 组
kind: Deployment
metadata:
  name: web
  namespace: dev
spec:
  replicas: 3
  selector:                # ← Deployment 用它找自己管的 Pod
    matchLabels:
      app: web
  template:                # ← Pod 模板，改这里会触发滚动更新
    metadata:
      labels:
        app: web           # ← 必须和上面 selector 匹配！
    spec:
      containers:
      - name: web
        image: nginx:1.27
```

**关键：** `spec.selector.matchLabels` 必须与 `spec.template.metadata.labels` 一致，否则创建报错。这也是「Service 选不到 Pod」类问题的根源。

## 三、滚动更新与回滚

### 3.1 更新镜像

```bash
kubectl set image deployment/web web=nginx:1.26 -n dev
```

| 部分 | 含义 |
|------|------|
| `set image deployment/web` | 对 web Deployment 改镜像 |
| `web=nginx:1.26` | **容器名=新镜像**（`web` 是容器名，不是 Deployment 名！） |

> **高频易错：** 等号左边是**容器名**。用 `kubectl get deploy web -o jsonpath='{.spec.template.spec.containers[*].name}'` 可查容器名。

### 3.2 查看更新过程与历史

```bash
kubectl rollout status deployment/web -n dev     # 阻塞等待更新完成
kubectl rollout history deployment/web -n dev    # 版本历史（每改一次模板 +1）
```

### 3.3 回滚

```bash
kubectl rollout undo deployment/web -n dev                    # 回上一版
kubectl rollout undo deployment/web -n dev --to-revision=2    # 回指定版本
```

> `undo` **不用指定镜像**，它用历史里保存的整份 Pod 模板。回滚前用 `rollout history --revision=N` 确认那一版镜像。

### 3.4 暂停 / 恢复

```bash
kubectl rollout pause deployment/web -n dev    # 暂停，改多处配置不逐次触发更新
kubectl rollout resume deployment/web -n dev   # 恢复，一次性应用
```

### 3.5 更新策略

```yaml
spec:
  strategy:
    type: RollingUpdate       # RollingUpdate（默认）| Recreate
    rollingUpdate:
      maxSurge: 1             # 更新时最多可超出期望副本数（多起几个）
      maxUnavailable: 0       # 更新时最多几个不可用（0 = 零停机）
```

| 类型 | 行为 |
|------|------|
| `RollingUpdate` | 逐个替换，服务不中断 |
| `Recreate` | 先全删旧的再建新的，有停机 |

## 四、ConfigMap 与 Secret

### 4.1 概念

| 对象 | 存什么 | 是否加密 |
|------|--------|----------|
| **ConfigMap** | 普通配置（环境名、日志级别、配置文件） | 明文 |
| **Secret** | 敏感信息（密码、token、证书） | 仅 Base64 编码，**不是加密** |

作用：把配置从镜像里**剥离**出来，改配置不用重打镜像。

### 4.2 创建

```bash
# ConfigMap：键值对
kubectl create configmap app-config \
  --from-literal=APP_MODE=production \
  --from-literal=LOG_LEVEL=info -n dev

# ConfigMap：从文件
kubectl create configmap nginx-conf --from-file=nginx.conf -n dev

# Secret：键值对
kubectl create secret generic db-creds \
  --from-literal=DB_USER=admin \
  --from-literal=DB_PASS=changeme -n dev
```

| 参数 | 含义 |
|------|------|
| `--from-literal=KEY=VALUE` | 直接给一个键值 |
| `--from-file=文件` | 用文件名作 key、文件内容作 value |
| `secret generic` | 通用 Secret（还有 `docker-registry`、`tls` 类型） |

### 4.3 三种注入方式

**① 全部键作为环境变量（envFrom）：**

```yaml
spec:
  containers:
  - name: app
    image: nginx
    envFrom:
    - configMapRef:
        name: app-config
    - secretRef:
        name: db-creds
```

**② 单个键作为环境变量（valueFrom）：**

```yaml
    env:
    - name: MODE                    # 容器里的变量名
      valueFrom:
        configMapKeyRef:
          name: app-config          # 来自哪个 ConfigMap
          key: APP_MODE             # 取哪个键
    - name: PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-creds
          key: DB_PASS
```

**③ 挂载为文件（volume）：**

```yaml
    volumeMounts:
    - name: config-vol
      mountPath: /etc/config        # 挂到容器里的目录
  volumes:
  - name: config-vol
    configMap:
      name: app-config              # 每个 key 变成一个文件
```

### 4.4 关键区别

| 注入方式 | 改了 ConfigMap 后 |
|----------|-------------------|
| 环境变量 | **不会自动更新**，需重启 Pod |
| 挂载为文件 | 会**自动更新**（有延迟） |

> 挂载卷时，若只想替换目录里某一个文件而不覆盖整个目录，用 `subPath`。

## 五、资源 requests / limits

### 5.1 概念

```yaml
    resources:
      requests:            # 「至少需要」——调度依据
        cpu: "250m"        # 250 毫核 = 0.25 个 CPU
        memory: "64Mi"
      limits:              # 「最多能用」——运行时上限
        cpu: "500m"
        memory: "128Mi"
```

| 字段 | 作用 | 超了会怎样 |
|------|------|-----------|
| `requests` | 调度器据此选节点（节点要有这么多空余） | — |
| `limits.cpu` | CPU 上限 | 被**限流**（throttle），不杀 |
| `limits.memory` | 内存上限 | 被 **OOMKilled**（杀掉重启） |

CPU 单位：`1` = 1 核，`500m` = 0.5 核。内存单位：`Mi`（1024 进制）、`M`（1000 进制）。

### 5.2 命名空间级约束

| 对象 | 作用 |
|------|------|
| **LimitRange** | 给命名空间里 Pod/容器设**默认值和上下限** |
| **ResourceQuota** | 限制整个命名空间的资源**总量** |

## 六、健康探针

让 K8s 知道容器「是否活着」「是否可以接流量」。

| 探针 | 作用 | 失败后果 |
|------|------|----------|
| **livenessProbe** | 存活探针，检测是否卡死 | 失败则**重启容器** |
| **readinessProbe** | 就绪探针，检测能否接流量 | 失败则从 Service **摘掉**，不重启 |
| **startupProbe** | 启动探针，给慢启动应用宽限期 | 失败则重启 |

```yaml
    livenessProbe:
      httpGet:
        path: /healthz
        port: 80
      initialDelaySeconds: 5      # 容器启动后等几秒才开始探测
      periodSeconds: 10           # 每隔几秒探一次
    readinessProbe:
      tcpSocket:
        port: 80
      periodSeconds: 5
```

探测方式：`httpGet`（HTTP 200-399 算成功）、`tcpSocket`（能连通算成功）、`exec`（命令退出 0 算成功）。

## 七、Pod 调度

控制 Pod 落在哪个节点。

### 7.1 nodeSelector（最简单）

先给节点打标签，再让 Pod 选：

```bash
kubectl label nodes k8s-worker1 disktype=ssd
```

```yaml
spec:
  nodeSelector:
    disktype: ssd        # 只调度到带此标签的节点
```

### 7.2 Node Affinity（更灵活）

```yaml
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:   # 硬性要求，必须满足
        nodeSelectorTerms:
        - matchExpressions:
          - key: disktype
            operator: In            # In / NotIn / Exists / DoesNotExist
            values: ["ssd"]
      preferredDuringSchedulingIgnoredDuringExecution:  # 软性偏好，尽量满足
      - weight: 1
        preference:
          matchExpressions:
          - key: zone
            operator: In
            values: ["zone-a"]
```

| 类型 | 含义 |
|------|------|
| `required...` | **必须**满足，否则不调度（硬约束） |
| `preferred...` | **尽量**满足，不满足也能调度（软约束） |

### 7.3 Taints 与 Tolerations（污点与容忍）

- **Taint（污点）**：打在**节点**上的「排斥」，没有对应容忍的 Pod 不能来。
- **Toleration（容忍）**：写在 **Pod** 上，表示「我能忍受这个污点」。

```bash
kubectl taint nodes k8s-worker1 key=value:NoSchedule     # 加污点
kubectl taint nodes k8s-worker1 key=value:NoSchedule-    # 末尾 - 表示移除
```

```yaml
spec:
  tolerations:
  - key: "key"
    operator: "Equal"
    value: "value"
    effect: "NoSchedule"
```

| Effect | 行为 |
|--------|------|
| `NoSchedule` | 新 Pod 不调度（已有的不动） |
| `PreferNoSchedule` | 尽量不调度 |
| `NoExecute` | 不调度 + **驱逐**已有 Pod |

> 记忆：**nodeSelector/Affinity 是 Pod「挑节点」；Taint 是节点「赶 Pod」；Toleration 是 Pod「我不怕」。**

### 7.4 手动指定节点（nodeName）

```yaml
spec:
  nodeName: k8s-worker2    # 直接绑死节点，跳过调度器
```

## 八、其他工作负载：DaemonSet / Job / CronJob / StatefulSet

| 类型 | 用途 | 特点 |
|------|------|------|
| **DaemonSet** | 每个节点跑一个 Pod | 日志/监控 Agent、CNI（如 calico-node） |
| **Job** | 一次性任务 | 跑到成功完成 |
| **CronJob** | 定时任务 | Cron 表达式触发 |
| **StatefulSet** | 有状态应用 | 稳定网络名、有序启停、独立存储 |

### 8.1 Job

```bash
kubectl create job pi --image=perl -- perl -Mbignum=bpi -wle 'print bpi(200)'
```

```yaml
spec:
  completions: 1        # 需要成功几次
  parallelism: 1        # 并行几个
  backoffLimit: 4       # 失败重试上限
  template:
    spec:
      restartPolicy: OnFailure   # Job 只能 OnFailure 或 Never
```

### 8.2 CronJob

```bash
kubectl create cronjob backup --image=busybox --schedule="*/5 * * * *" -- echo hello
```

| 字段 | 含义 |
|------|------|
| `--schedule="*/5 * * * *"` | 每 5 分钟执行一次（标准 cron 五段） |

cron 五段：`分 时 日 月 周`。

### 8.3 DaemonSet

DaemonSet 没有 `kubectl create daemonset` 命令，用 YAML 或从 Deployment 改：把 `kind: Deployment` 改成 `kind: DaemonSet`，删掉 `replicas`（每节点固定一个）。

## 九、静态 Pod

### 9.1 概念

**静态 Pod** 由**节点上的 kubelet 直接管理**，不经过 API Server、也不受调度器控制。

- manifest 放在节点目录 `/etc/kubernetes/manifests/`
- kubelet 监视该目录，有 YAML 就起对应 Pod
- 控制面组件（apiserver、etcd、scheduler、controller-manager）就是静态 Pod
- 在 `kubectl get pods` 里能看到，名字带节点后缀（如 `kube-apiserver-k8s-master`）
- **不能用 `kubectl delete` 删除**，删了会被 kubelet 重建；要删就删 manifest 文件

### 9.2 创建静态 Pod

```bash
# 1. 确认 manifest 目录
sudo grep staticPodPath /var/lib/kubelet/config.yaml
# 默认输出：staticPodPath: /etc/kubernetes/manifests

# 2. 在该目录放 Pod YAML（在目标节点上操作）
sudo tee /etc/kubernetes/manifests/static-web.yaml > /dev/null <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: static-web
spec:
  containers:
  - name: web
    image: nginx:1.27
EOF

# kubelet 会自动创建，几秒后：
kubectl get pods    # 看到 static-web-<节点名>
```

删除：`sudo rm /etc/kubernetes/manifests/static-web.yaml`。

## 十、节点维护

```bash
kubectl cordon k8s-worker1        # 标记不可调度（新 Pod 不来，老 Pod 不动）
kubectl drain k8s-worker1 --ignore-daemonsets --delete-emptydir-data  # 驱逐 Pod
kubectl uncordon k8s-worker1      # 恢复可调度
```

| 参数 | 含义 |
|------|------|
| `cordon` | 只封锁，不驱逐 |
| `drain` | 封锁 + 把 Pod 赶到别的节点 |
| `--ignore-daemonsets` | DaemonSet 的 Pod 无法迁移，忽略它们（否则 drain 报错） |
| `--delete-emptydir-data` | 允许删除用了 emptyDir 的 Pod（否则 drain 卡住） |
| `uncordon` | 解除封锁 |

> 这三个命令在 **kubeadm 升级工作节点** 时也会用到（见第五部分）。

## 十二、易错点与自检清单

### 易错点

- `set image` 等号左边是**容器名**，不是 Deployment 名。
- Deployment 的 `selector.matchLabels` 必须与 `template.labels` 一致。
- 环境变量方式注入 ConfigMap，改配置后**必须重启 Pod** 才生效。
- 静态 Pod **不能** `kubectl delete`，要删 manifest 文件。
- Job 的 `restartPolicy` 只能是 `OnFailure` 或 `Never`。
- `drain` 记得加 `--ignore-daemonsets`，否则报错。
