## 一、Kubernetes 架构

### 1.1 控制平面组件（Control Plane）

跑在 master 节点，负责「大脑」决策。

| 组件 | 职责 |
|------|------|
| **kube-apiserver** | 集群 API 入口，**唯一**能读写 etcd 的组件 |
| **etcd** | 分布式键值数据库，保存集群**全部状态** |
| **kube-scheduler** | 决定 Pod 调度到哪个节点 |
| **kube-controller-manager** | 运行各类控制器（Deployment、Node 等），让实际状态趋近期望状态 |
| **cloud-controller-manager** | 对接云厂商（考试通常不涉及） |

### 1.2 工作节点组件（Worker Node）

每个节点都有，负责「干活」。

| 组件 | 职责 |
|------|------|
| **kubelet** | 管理本节点 Pod 生命周期，向 API Server 汇报（**systemd 服务，不是 Pod**） |
| **kube-proxy** | 维护 Service 转发规则（iptables/IPVS） |
| **容器运行时** | containerd，真正拉镜像、跑容器（通过 CRI 接口） |

### 1.3 关键概念

- kubeadm 集群里，**控制面组件以静态 Pod 运行**，manifest 在 `/etc/kubernetes/manifests/`。
- **kubelet 是 systemd 服务**，用 `systemctl status kubelet` 管理，不是 Pod。
- 三大扩展接口：**CRI**（运行时）、**CNI**（网络）、**CSI**（存储）。

请求流转：

```
kubectl → kube-apiserver → etcd（存）
                        ↘ scheduler / controller-manager（决策）
                        ↘ 各节点 kubelet（执行）→ containerd → 容器
```

## 二、RBAC 权限控制

### 2.1 概念

**RBAC（Role-Based Access Control）** 控制「**谁（主体）** 能对 **哪些资源** 做 **什么操作**」。

四个核心对象：

| 对象 | 作用域 | 说明 |
|------|--------|------|
| **Role** | 命名空间 | 定义命名空间内的权限 |
| **ClusterRole** | 集群 | 定义集群级权限（或跨命名空间复用） |
| **RoleBinding** | 命名空间 | 把 Role/ClusterRole **绑定**给主体，权限只在该命名空间生效 |
| **ClusterRoleBinding** | 集群 | 把 ClusterRole 绑定给主体，**全集群**生效 |

**主体（Subject）三种：** User（用户）、Group（组）、**ServiceAccount**（给 Pod 用的账号）。

**权限 = verbs + resources：**

- verbs（动作）：`get`、`list`、`watch`、`create`、`update`、`patch`、`delete`
- resources（资源）：`pods`、`deployments`、`services`…

### 2.2 关键组合规则（易考）

| 组合 | 生效范围 |
|------|----------|
| Role + RoleBinding | 单个命名空间 |
| ClusterRole + RoleBinding | **仅该 RoleBinding 所在命名空间**（用 ClusterRole 复用定义） |
| ClusterRole + ClusterRoleBinding | 全集群 |

### 2.3 命令式创建（考试首选，快）

```bash
# 1. 建命名空间内的 Role：能读 pods
kubectl create role pod-reader --verb=get,list,watch --resource=pods -n dev

# 2. 建 ServiceAccount
kubectl create serviceaccount my-sa -n dev

# 3. 建 RoleBinding：把 Role 绑给 SA
kubectl create rolebinding read-pods \
  --role=pod-reader \
  --serviceaccount=dev:my-sa \
  -n dev

# 4. 集群级：ClusterRole + ClusterRoleBinding（读节点）
kubectl create clusterrole node-reader --verb=get,list --resource=nodes
kubectl create clusterrolebinding read-nodes \
  --clusterrole=node-reader \
  --serviceaccount=dev:my-sa
```

| 参数 | 含义 |
|------|------|
| `--verb=get,list,watch` | 允许的动作 |
| `--resource=pods` | 作用的资源 |
| `--role=pod-reader` | 绑定哪个 Role |
| `--serviceaccount=dev:my-sa` | 主体：`命名空间:SA名` |
| `--clusterrole=` | 绑定哪个 ClusterRole |

### 2.4 验证权限（考试必做）

```bash
kubectl auth can-i list pods -n dev --as=system:serviceaccount:dev:my-sa
kubectl auth can-i list nodes --as=system:serviceaccount:dev:my-sa
kubectl auth can-i create deployments --as=john -n prod
```

| 部分 | 含义 |
|------|------|
| `auth can-i` | 检查某主体能否做某操作 |
| `--as=system:serviceaccount:dev:my-sa` | 模拟这个 ServiceAccount |
| `--as=john` | 模拟用户 john |

返回 `yes` / `no`。**每道 RBAC 题做完都用它验证。**

### 2.5 YAML 形式

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: dev
rules:
- apiGroups: [""]              # "" 表示核心组（pods/services 等）
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: dev
subjects:
- kind: ServiceAccount
  name: my-sa
  namespace: dev
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

> `apiGroups: [""]` 是核心组；`deployments` 属于 `apps` 组，要写 `apiGroups: ["apps"]`。

## 三、kubeadm 集群安装回顾

你在第一部分已完整做过，这里提炼考试可能考的**节点前置条件修复**：

```bash
swapoff -a                                   # 关闭 swap（kubelet 要求）
modprobe overlay && modprobe br_netfilter    # 加载内核模块
# sysctl 需含：net.bridge.bridge-nf-call-iptables = 1、net.ipv4.ip_forward = 1
sysctl --system
systemctl status containerd                  # 容器运行时正常
systemctl enable --now kubelet               # kubelet 开机自启并启动
```

**worker 加入：**

```bash
sudo kubeadm join <master-ip>:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash>

# 忘记/token 过期，在 master 重新生成完整 join 命令
kubeadm token create --print-join-command
```

| 参数 | 含义 |
|------|------|
| `<master-ip>:6443` | API Server 地址与端口 |
| `--token` | 一次性引导令牌（默认 24h 过期） |
| `--discovery-token-ca-cert-hash` | 校验 master 身份，防中间人 |

## 四、kubeadm 集群升级（高频）

### 4.1 核心记忆点

> **控制面节点用 `kubeadm upgrade apply`，工作节点用 `kubeadm upgrade node`。**  
> 升级工作节点前，先在控制面 `drain` 它。

升级顺序：**先控制面，再逐个工作节点**。每个组件版本要一致。

### 4.2 控制平面节点升级

```bash
# 1. 解锁并升级 kubeadm 本身
sudo apt-mark unhold kubeadm
sudo apt-get update && sudo apt-get install -y kubeadm=1.35.0-1.1
sudo apt-mark hold kubeadm

# 2. 查看升级计划（会列出可升到的版本）
sudo kubeadm upgrade plan

# 3. 执行升级（控制面专用命令）
sudo kubeadm upgrade apply v1.35.0

# 4. drain 控制面节点（腾空工作负载）
kubectl drain <cp-node> --ignore-daemonsets

# 5. 升级 kubelet 和 kubectl
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.35.0-1.1 kubectl=1.35.0-1.1
sudo apt-mark hold kubelet kubectl

# 6. 重启 kubelet
sudo systemctl daemon-reload
sudo systemctl restart kubelet

# 7. 恢复调度
kubectl uncordon <cp-node>
```

| 命令 | 作用 |
|------|------|
| `apt-mark unhold/hold` | 解锁/锁定版本（平时锁定防误升级） |
| `kubeadm upgrade plan` | 显示升级计划和目标版本 |
| `kubeadm upgrade apply vX` | **控制面**执行升级 |
| `drain` | 驱逐节点 Pod（见第二部分） |
| `daemon-reload` + `restart kubelet` | 让新版 kubelet 生效 |

### 4.3 工作节点升级

```bash
# 1. 在控制面 drain 该工作节点
kubectl drain <worker-node> --ignore-daemonsets --delete-emptydir-data

# 2. SSH 到工作节点，升级 kubeadm
sudo apt-mark unhold kubeadm
sudo apt-get update && sudo apt-get install -y kubeadm=1.35.0-1.1
sudo apt-mark hold kubeadm

# 3. 执行节点升级（工作节点专用命令）
sudo kubeadm upgrade node

# 4. 升级 kubelet、kubectl
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.35.0-1.1 kubectl=1.35.0-1.1
sudo apt-mark hold kubelet kubectl

# 5. 重启 kubelet
sudo systemctl daemon-reload
sudo systemctl restart kubelet

# 6. 回控制面恢复调度
kubectl uncordon <worker-node>
```

> **区别只在第 3 步：** 工作节点是 `kubeadm upgrade node`，没有 `apply`。

### 4.4 验证

```bash
kubectl get nodes         # VERSION 列应显示新版本
```

## 五、etcd 备份与恢复（几乎必考）

etcd 存着集群所有数据，备份恢复是**运维底线**，CKA 高频。

### 5.1 先找证书路径（关键，别背死）

证书路径从 etcd 静态 Pod 的 manifest 里查，**不同环境可能不同**：

```bash
sudo cat /etc/kubernetes/manifests/etcd.yaml | grep -E "cert-file|key-file|trusted-ca-file|listen-client-urls|data-dir"
```

你会看到类似：

```
--cert-file=/etc/kubernetes/pki/etcd/server.crt
--key-file=/etc/kubernetes/pki/etcd/server.key
--trusted-ca-file=/etc/kubernetes/pki/etcd/ca.crt
--listen-client-urls=https://127.0.0.1:2379,...
--data-dir=/var/lib/etcd
```

> **考试失分点第一名：证书路径写错。** 一定从 etcd.yaml 里复制，不要凭记忆。

### 5.2 备份（snapshot save）

```bash
ETCDCTL_API=3 etcdctl snapshot save /tmp/etcd-backup.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
```

| 参数 | 含义 |
|------|------|
| `ETCDCTL_API=3` | 用 etcdctl v3 API（必须） |
| `snapshot save 路径` | 备份保存到哪 |
| `--endpoints` | etcd 地址（本机 2379） |
| `--cacert` | CA 证书（校验服务端） |
| `--cert` / `--key` | 客户端证书和私钥（证明自己身份） |

### 5.3 验证备份

```bash
ETCDCTL_API=3 etcdctl snapshot status /tmp/etcd-backup.db --write-out=table
```

显示备份的 key 数、大小等，能看到内容说明备份有效。

### 5.4 恢复（snapshot restore）

```bash
# 1. 恢复快照到一个新目录（不要覆盖原 data-dir）
ETCDCTL_API=3 etcdctl snapshot restore /tmp/etcd-backup.db \
  --data-dir=/var/lib/etcd-restored
```

> **新版注意：** 较新 etcd 里 `etcdctl snapshot restore` 会提示已弃用，官方推荐用 **`etcdutl`**（备份仍用 `etcdctl snapshot save`）。两者二选一，效果相同：
>
> ```bash
> etcdutl snapshot restore /tmp/etcd-backup.db --data-dir=/var/lib/etcd-restored
> ```
> 考试环境用哪个都可以；若 `etcdctl restore` 报弃用错误，改用 `etcdutl`。

```bash
# 2. 修改 etcd 静态 Pod manifest 指向新目录
sudo vi /etc/kubernetes/manifests/etcd.yaml
```

要改两处，保持一致：

```yaml
    - --data-dir=/var/lib/etcd-restored     # ① 启动参数
    volumes:
    - hostPath:
        path: /var/lib/etcd-restored        # ② hostPath 挂载
      name: etcd-data
```

```bash
# 3. kubelet 检测到 manifest 变化会自动重启 etcd
#    期间 apiserver/kubectl 可能短暂不可用，属正常，等一会
kubectl get pods -n kube-system    # 恢复后验证
```

| 步骤 | 要点 |
|------|------|
| restore | 恢复到**新目录**，不要直接覆盖 |
| 改 manifest | `--data-dir` 和 `hostPath.path` 两处都要改成新目录 |
| 等待 | 静态 Pod 自动重启，无需手动 `kubectl apply` |

### 5.5 etcd 健康检查

```bash
ETCDCTL_API=3 etcdctl endpoint health \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
```

## 六、证书管理

kubeadm 集群证书默认 1 年有效期。

```bash
sudo kubeadm certs check-expiration     # 查看所有证书到期时间
sudo kubeadm certs renew all            # 续期所有证书
# 续期后重启受影响的控制面组件（删对应静态 Pod 让其重建，或重启 kubelet）
```

| 命令 | 作用 |
|------|------|
| `certs check-expiration` | 列出证书到期时间 |
| `certs renew all` | 续期全部证书 |

## 七、高可用（HA）控制平面

考试主要要求**理解概念**。

### 7.1 两种 etcd 拓扑

| 模式 | 说明 |
|------|------|
| **Stacked etcd** | etcd 与控制面组件同节点（kubeadm 默认） |
| **External etcd** | etcd 独立集群，隔离性更好 |

### 7.2 HA 要点

- 多个控制面节点 + 前面一个负载均衡器（LB）
- 初始化时指定 LB 地址：

```bash
kubeadm init --control-plane-endpoint=<LB-IP>:6443 --upload-certs
```

- 其他控制面节点加入：

```bash
kubeadm join <LB-IP>:6443 --control-plane --certificate-key <key> ...
```

| 参数 | 含义 |
|------|------|
| `--control-plane-endpoint` | 稳定的 LB 入口（HA 必需） |
| `--upload-certs` | 上传证书供其他控制面共享 |
| `--control-plane` | 以控制面身份加入（而非 worker） |

## 八、Helm 与 Kustomize

### 8.1 Helm（Kubernetes 包管理器）

把一组资源打包成「Chart」，一条命令安装。

```bash
helm repo add bitnami https://charts.bitnami.com/bitnami   # 加仓库
helm repo update                                           # 更新索引
helm install my-nginx bitnami/nginx -n default             # 安装
helm list -A                                               # 列出已安装 release
helm upgrade my-nginx bitnami/nginx --set replicaCount=3   # 升级/改参数
helm uninstall my-nginx                                    # 卸载
```

| 命令 | 作用 |
|------|------|
| `repo add/update` | 管理 Chart 仓库 |
| `install <名> <chart>` | 安装一个 release |
| `--set key=value` | 覆盖 Chart 默认值 |
| `list -A` | 所有命名空间的 release |

**只渲染不安装（2026 高频题型）：**

考试可能要求「用 Helm **生成 manifest** 而不真正安装」，如指定版本、禁用 CRD：

```bash
helm template my-app bitnami/nginx \
  --version 15.0.0 \
  --set replicaCount=2 > out.yaml       # 渲染成 YAML 保存，不部署

helm search repo nginx --versions        # 查可用的 chart 版本
helm show values bitnami/nginx           # 查看 chart 可配置项
```

| 参数 | 含义 |
|------|------|
| `helm template` | 本地渲染 Chart 为 YAML，**不连集群、不安装** |
| `--version` | 指定 Chart 版本（注意是 Chart 版本，不是应用版本） |
| `--set` / `-f values.yaml` | 覆盖默认值 |
| `--include-crds` / `--skip-crds` | 是否包含/跳过 CRD |

### 8.2 Kustomize（kubectl 内置）

不改原始 YAML，用「叠加（overlay）」方式定制。

```bash
kubectl apply -k ./overlays/production    # 应用某个 overlay
kubectl kustomize ./base                  # 预览渲染结果（不应用）
```

| 参数 | 含义 |
|------|------|
| `-k 目录` | 使用该目录的 kustomization.yaml |
| `kubectl kustomize` | 只渲染不应用，用于预览 |

## 九、CRD 与 Operator

- **CRD（CustomResourceDefinition）**：扩展 Kubernetes API，注册**自定义资源类型**（比如 `IPPool`）。
- **Operator**：CRD + 自定义控制器，把「运维知识」代码化，自动管理复杂应用。
- 考试要求：**理解概念**，能识别/查看 CRD 资源，可能涉及用 Helm 装个 Operator。

```bash
kubectl get crd                       # 列出所有自定义资源类型
kubectl api-resources                 # 查所有资源类型（含 CRD）
kubectl get <custom-resource> -A      # 查看某自定义资源
```

## 十一、易错点与自检清单

### 易错点

- etcd 证书路径**从 etcd.yaml 复制**，别背。
- etcd 恢复后要改 manifest 的 **两处**（`--data-dir` 和 `hostPath.path`）。
- 升级：控制面 `apply`，工作节点 `node`；工作节点先 `drain`。
- 升级前后 `apt-mark unhold`/`hold` 别漏。
- RBAC：ClusterRole + RoleBinding = 只在该命名空间生效。
- 每道 RBAC 题用 `auth can-i` 验证。
