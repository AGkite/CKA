## 一、Kubernetes 架构

### 1.1 控制平面组件（Control Plane）

跑在 master 节点，负责「大脑」决策。

| 组件 | 职责 |
|------|------|
| **kube-apiserver** | 集群 API 入口，**唯一**能读写 etcd 的组件 |
| **etcd** | 分布式键值数据库，保存集群**全部状态** |
| **kube-scheduler** | 决定 Pod 调度到哪个节点 |
| **kube-controller-manager** | 运行各类控制器（Deployment、Node 等），让实际状态趋近期望状态 |
| **cloud-controller-manager** | 对接云厂商（考试通常不涉及） |

### 1.2 工作节点组件（Worker Node）

每个节点都有，负责「干活」。

| 组件 | 职责 |
|------|------|
| **kubelet** | 管理本节点 Pod 生命周期，向 API Server 汇报（**systemd 服务，不是 Pod**） |
| **kube-proxy** | 维护 Service 转发规则（iptables/IPVS） |
| **容器运行时** | containerd，真正拉镜像、跑容器（通过 CRI 接口） |

### 1.3 关键概念

- kubeadm 集群里，**控制面组件以静态 Pod 运行**，manifest 在 `/etc/kubernetes/manifests/`。
- **kubelet 是 systemd 服务**，用 `systemctl status kubelet` 管理，不是 Pod。
- 三大扩展接口：**CRI**（运行时）、**CNI**（网络）、**CSI**（存储）。

请求流转：

**执行位置：概念图，不执行。**

```
kubectl → kube-apiserver → etcd（存）
                        ↘ scheduler / controller-manager（决策）
                        ↘ 各节点 kubelet（执行）→ containerd → 容器
```

## 二、RBAC 权限控制

### 2.1 概念

**RBAC（Role-Based Access Control）** 控制「**谁（主体）** 能对 **哪些资源** 做 **什么操作**」。

四个核心对象：

| 对象 | 作用域 | 说明 |
|------|--------|------|
| **Role** | 命名空间 | 定义命名空间内的权限 |
| **ClusterRole** | 集群 | 定义集群级权限（或跨命名空间复用） |
| **RoleBinding** | 命名空间 | 把 Role/ClusterRole **绑定**给主体，权限只在该命名空间生效 |
| **ClusterRoleBinding** | 集群 | 把 ClusterRole 绑定给主体，**全集群**生效 |

**主体（Subject）三种：** User（用户）、Group（组）、**ServiceAccount**（给 Pod 用的账号）。

**权限 = verbs + resources：**

- verbs（动作）：`get`、`list`、`watch`、`create`、`update`、`patch`、`delete`
- resources（资源）：`pods`、`deployments`、`services`…

### 2.2 关键组合规则（易考）

| 组合 | 生效范围 |
|------|----------|
| Role + RoleBinding | 单个命名空间 |
| ClusterRole + RoleBinding | **仅该 RoleBinding 所在命名空间**（用 ClusterRole 复用定义） |
| ClusterRole + ClusterRoleBinding | 全集群 |

### 2.3 命令式创建（考试首选，快）

**执行位置：控制端 WSL。**

```bash
# 1. 建命名空间内的 Role：能读 pods
kubectl create role pod-reader --verb=get,list,watch --resource=pods -n dev

# 2. 建 ServiceAccount
kubectl create serviceaccount my-sa -n dev

# 3. 建 RoleBinding：把 Role 绑给 SA
kubectl create rolebinding read-pods \
  --role=pod-reader \
  --serviceaccount=dev:my-sa \
  -n dev

# 4. 集群级：ClusterRole + ClusterRoleBinding（读节点）
kubectl create clusterrole node-reader --verb=get,list --resource=nodes
kubectl create clusterrolebinding read-nodes \
  --clusterrole=node-reader \
  --serviceaccount=dev:my-sa
```

| 参数 | 含义 |
|------|------|
| `--verb=get,list,watch` | 允许的动作 |
| `--resource=pods` | 作用的资源 |
| `--role=pod-reader` | 绑定哪个 Role |
| `--serviceaccount=dev:my-sa` | 主体：`命名空间:SA名` |
| `--clusterrole=` | 绑定哪个 ClusterRole |

### 2.4 验证权限（考试必做）

**执行位置：控制端 WSL。**

```bash
kubectl auth can-i list pods -n dev --as=system:serviceaccount:dev:my-sa
kubectl auth can-i list nodes --as=system:serviceaccount:dev:my-sa
kubectl auth can-i create deployments --as=john -n prod
```

| 部分 | 含义 |
|------|------|
| `auth can-i` | 检查某主体能否做某操作 |
| `--as=system:serviceaccount:dev:my-sa` | 模拟这个 ServiceAccount |
| `--as=john` | 模拟用户 john |

返回 `yes` / `no`。**每道 RBAC 题做完都用它验证。**

### 2.5 YAML 形式

**执行位置：控制端 WSL；保存为 YAML 后用 `kubectl apply -f <文件>` 提交。**

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: dev
rules:
- apiGroups: [""]              # "" 表示核心组（pods/services 等）
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: dev
subjects:
- kind: ServiceAccount
  name: my-sa
  namespace: dev
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

> `apiGroups: [""]` 是核心组；`deployments` 属于 `apps` 组，要写 `apiGroups: ["apps"]`。

## 三、kubeadm 集群安装回顾

你在第一部分已完整做过，这里提炼考试可能考的**节点前置条件修复**：

**执行位置：目标节点宿主机；三台节点分别执行。**

```bash
sudo swapoff -a                              # 关闭 swap（kubelet 要求）
sudo modprobe overlay                        # 加载容器网络所需模块
sudo modprobe br_netfilter
# sysctl 需含：net.bridge.bridge-nf-call-iptables = 1、net.ipv4.ip_forward = 1
sudo sysctl --system
sudo systemctl status containerd             # 容器运行时正常
sudo systemctl enable --now kubelet          # kubelet 开机自启并启动
```

**在控制平面生成 worker 加入命令：**

**执行位置：`k8s-master` 宿主机。**

```bash
sudo kubeadm token create --print-join-command
```

复制输出后再到待加入节点执行，不能把两台机器的命令混在同一个命令块里。

**执行位置：尚未加入集群的目标 worker 宿主机（`k8s-worker1` 或 `k8s-worker2`），每台分别执行。**

```bash
sudo kubeadm join 10.13.31.10:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash>
```

| 参数 | 含义 |
|------|------|
| `<master-ip>:6443` | API Server 地址与端口 |
| `--token` | 一次性引导令牌（默认 24h 过期） |
| `--discovery-token-ca-cert-hash` | 校验 master 身份，防中间人 |

## 四、kubeadm 集群升级（高频）

> **高风险：仅可重建环境选做。** 升级会依次驱逐工作负载并重启节点组件。先确认集群可重建、etcd 快照可用，再动手。

### 4.1 核心记忆点

> **控制面节点用 `kubeadm upgrade apply`，工作节点用 `kubeadm upgrade node`。**  
> 升级工作节点前，先在控制面 `drain` 它。

升级顺序：**先控制面，再逐个工作节点**。练习时坚持**一次只升一个 minor**（例如 `1.34.x → 1.35.x`），不要跨 minor 跳升。具体补丁版本和 Debian 包版本必须以现场探测结果为准。

### 4.2 升级前探测版本与 apt 候选

**执行位置：控制端 WSL。**

```bash
kubectl get nodes -o wide
kubectl version
```

**执行位置：`k8s-master` 宿主机。**

```bash
kubeadm version -o short
sudo apt-get update
apt-cache policy kubeadm
apt-cache madison kubeadm
```

从 `apt-cache madison` 输出中选择“当前 minor 的下一个 minor”的具体包版本。下文用 `<目标包版本>`（如 `1.35.1-1.1`）和 `<目标K8s版本>`（如 `v1.35.1`）占位，两者必须对应。

### 4.3 控制平面节点升级

**执行位置：`k8s-master` 宿主机。**

```bash
# 1. 只升级 kubeadm，并再次确认实际版本
sudo apt-mark unhold kubeadm
sudo apt-get update
sudo apt-get install -y kubeadm=<目标包版本>
sudo apt-mark hold kubeadm
kubeadm version -o short

# 2. 先看计划，再升级控制平面
sudo kubeadm upgrade plan
sudo kubeadm upgrade apply <目标K8s版本>
```

**执行位置：控制端 WSL。**

```bash
# 3. 腾空控制平面节点
kubectl drain k8s-master --ignore-daemonsets
```

**执行位置：`k8s-master` 宿主机。**

```bash
# 4. 升级本机 kubelet、kubectl 并重启 kubelet
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=<目标包版本> kubectl=<目标包版本>
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload
sudo systemctl restart kubelet
```

**执行位置：控制端 WSL。**

```bash
# 5. 确认节点恢复后再解除不可调度
kubectl get nodes
kubectl uncordon k8s-master
```

| 命令 | 作用 |
|------|------|
| `apt-mark unhold/hold` | 解锁/锁定版本（平时锁定防误升级） |
| `kubeadm upgrade plan` | 显示升级计划和目标版本 |
| `kubeadm upgrade apply vX` | **控制面**执行升级 |
| `drain` | 驱逐节点 Pod（见第二部分） |
| `daemon-reload` + `restart kubelet` | 让新版 kubelet 生效 |

### 4.4 工作节点升级

每次只处理一个 worker。下面先以 `k8s-worker1` 为例，验证完成后再把节点名换成 `k8s-worker2` 重复。

**执行位置：控制端 WSL。**

```bash
# 1. 先腾空目标节点
kubectl drain k8s-worker1 --ignore-daemonsets --delete-emptydir-data
```

> `--delete-emptydir-data` 会随驱逐删除 Pod 的 `emptyDir` 临时数据。执行前检查该节点上的 Pod、PDB 和本地临时数据，确认允许丢弃后再使用。

**执行位置：`k8s-worker1` 宿主机（目标节点）。**

```bash
# 2. 升级 kubeadm，再执行 worker 专用升级
sudo apt-mark unhold kubeadm
sudo apt-get update
sudo apt-get install -y kubeadm=<目标包版本>
sudo apt-mark hold kubeadm
kubeadm version -o short
sudo kubeadm upgrade node

# 3. 升级 kubelet、kubectl 并重启 kubelet
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=<目标包版本> kubectl=<目标包版本>
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload
sudo systemctl restart kubelet
```

**执行位置：控制端 WSL。**

```bash
# 4. 等 k8s-worker1 Ready 后恢复调度，再处理 k8s-worker2
kubectl get nodes
kubectl uncordon k8s-worker1
```

> **角色区别：** 首个控制平面执行 `kubeadm upgrade apply <版本>`；工作节点执行 `kubeadm upgrade node`，没有 `apply`。

### 4.5 验证

**执行位置：控制端 WSL。**

```bash
kubectl get nodes -o wide
kubectl get pods -A
kubectl get events -A --sort-by='.lastTimestamp'
```

## 五、etcd 备份与恢复（几乎必考）

etcd 存着集群所有数据，备份恢复是**运维底线**，CKA 高频。

### 5.1 先找证书路径（关键，别背死）

证书路径从 etcd 静态 Pod 的 manifest 里查，**不同环境可能不同**：

**执行位置：`k8s-master` 宿主机。**

```bash
sudo grep -E "cert-file|key-file|trusted-ca-file|listen-client-urls|data-dir" \
  /etc/kubernetes/manifests/etcd.yaml
```

你会看到类似：

**执行位置：示例输出，不执行。**

```
--cert-file=/etc/kubernetes/pki/etcd/server.crt
--key-file=/etc/kubernetes/pki/etcd/server.key
--trusted-ca-file=/etc/kubernetes/pki/etcd/ca.crt
--listen-client-urls=https://127.0.0.1:2379,...
--data-dir=/var/lib/etcd
```

> `server.crt/key` 是 etcd 服务端身份。备份和健康检查使用专用客户端凭据 `healthcheck-client.crt/key`；路径仍要在现场确认。

### 5.2 备份（snapshot save）

快照不要放 `/tmp`。先建持久目录，再备份并立即校验。

**执行位置：`k8s-master` 宿主机。**

```bash
sudo install -d -m 0700 /var/backups/etcd
sudo ETCDCTL_API=3 etcdctl snapshot save /var/backups/etcd/snapshot.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
  --key=/etc/kubernetes/pki/etcd/healthcheck-client.key

sudo etcdutl snapshot status /var/backups/etcd/snapshot.db --write-out=table
```

| 参数 | 含义 |
|------|------|
| `ETCDCTL_API=3` | 用 etcdctl v3 API（必须） |
| `snapshot save 路径` | 备份保存到哪 |
| `--endpoints` | etcd 地址（本机 2379） |
| `--cacert` | CA 证书（校验服务端） |
| `--cert` / `--key` | 客户端证书和私钥（证明自己身份） |

### 5.3 验证备份

若环境没有 `etcdutl`，可用兼容命令再次验证：

**执行位置：`k8s-master` 宿主机。**

```bash
sudo ETCDCTL_API=3 etcdctl snapshot status \
  /var/backups/etcd/snapshot.db --write-out=table
```

显示备份的 key 数、大小等，能看到内容说明备份有效。

### 5.4 恢复（snapshot restore）

> **高风险：仅可重建环境选做。** 恢复会替换集群状态，并导致 API Server 暂时不可用。先确认快照校验成功，并保留原 manifest 与原数据目录用于回滚。

**执行位置：`k8s-master` 宿主机。**

```bash
# 1. manifest 备份必须放在 manifests 目录外，否则 kubelet 会把备份也当静态 Pod
sudo install -d -m 0700 /var/backups/kubernetes-manifests
sudo cp -a /etc/kubernetes/manifests/etcd.yaml \
  /var/backups/kubernetes-manifests/etcd.yaml.before-restore

# 2. 恢复到新的宿主机目录，不覆盖 /var/lib/etcd。
# name、peer URL 和 initial-cluster 必须与 etcd.yaml 中的现有参数一致。
sudo rm -rf /var/lib/etcd-restored
sudo etcdutl snapshot restore /var/backups/etcd/snapshot.db \
  --data-dir=/var/lib/etcd-restored \
  --name=k8s-master \
  --initial-advertise-peer-urls=https://10.13.31.10:2380 \
  --initial-cluster=k8s-master=https://10.13.31.10:2380
```

> 新版优先使用 `etcdutl snapshot restore`；备份仍使用 `etcdctl snapshot save`。执行前应从 `/etc/kubernetes/manifests/etcd.yaml` 核对 `--name`、`--initial-advertise-peer-urls` 和 `--initial-cluster`，不要在其他集群机械照抄本环境的值。

**执行位置：`k8s-master` 宿主机。**

```bash
# 3. 只修改 hostPath.path；容器内 mountPath 和 --data-dir 保持 /var/lib/etcd
sudo vi /etc/kubernetes/manifests/etcd.yaml
```

目标片段应为：

**执行位置：`k8s-master` 上 etcd 静态 Pod manifest 的目标配置，不是独立执行命令。**

```yaml
    - --data-dir=/var/lib/etcd
    volumeMounts:
    - mountPath: /var/lib/etcd
      name: etcd-data
    volumes:
    - hostPath:
        path: /var/lib/etcd-restored
        type: DirectoryOrCreate
      name: etcd-data
```

即宿主机 `/var/lib/etcd-restored` 映射到容器 `/var/lib/etcd`，容器启动参数仍是 `--data-dir=/var/lib/etcd`。不要把宿主机恢复目录直接写成容器内 data-dir。

**执行位置：控制端 WSL；等待静态 Pod 自动重建后执行。**

```bash
kubectl get --raw='/readyz?verbose'
kubectl get nodes
kubectl get pods -A
```

| 步骤 | 要点 |
|------|------|
| restore | 恢复到**新目录**，不要直接覆盖 |
| 改 manifest | `hostPath.path` 改为宿主机恢复目录；`mountPath` 与 `--data-dir` 保持容器内 `/var/lib/etcd` |
| 等待 | 静态 Pod 自动重启，无需手动 `kubectl apply` |

**失败回滚：** 如果 API 长时间无法恢复，先从节点本地日志确认问题，再恢复原 manifest。不要删除原 `/var/lib/etcd`。

**执行位置：`k8s-master` 宿主机。**

```bash
sudo crictl ps -a --name etcd
sudo crictl logs <etcd-container-id>
sudo journalctl -u kubelet --since "-10 min" --no-pager

sudo cp -a /var/backups/kubernetes-manifests/etcd.yaml.before-restore \
  /etc/kubernetes/manifests/etcd.yaml
```

**执行位置：控制端 WSL；回滚后验证。**

```bash
kubectl get --raw='/readyz?verbose'
kubectl get nodes
kubectl get pods -A
```

### 5.5 etcd 健康检查

**执行位置：`k8s-master` 宿主机。**

```bash
sudo ETCDCTL_API=3 etcdctl endpoint health \
  --endpoints=https://127.0.0.1:2379 \
```

## 六、证书管理

kubeadm 生成的大多数非 CA 叶子证书默认 1 年有效；CA 证书通常更长，必须以现场检查结果为准。

**执行位置：`k8s-master` 宿主机；检查操作只读。**

```bash
sudo kubeadm certs check-expiration
```

| 命令 | 作用 |
|------|------|
| `certs check-expiration` | 列出证书到期时间 |
| `certs renew all` | 续期全部证书 |

> **高风险：证书续期仅可重建环境选做。** `renew all` 后，已运行的静态 Pod 不会自动加载新证书，必须逐个触发重建；控制端旧 kubeconfig 也不会自动更新。

**执行位置：`k8s-master` 宿主机。**

```bash
sudo kubeadm certs renew all

# 临时移出再移回，每次只处理一个 manifest，并等待对应容器恢复
sudo install -d -m 0700 /var/backups/static-pod-restart
for manifest in kube-apiserver.yaml kube-controller-manager.yaml kube-scheduler.yaml etcd.yaml; do
  sudo mv "/etc/kubernetes/manifests/${manifest}" /var/backups/static-pod-restart/
  sleep 25
  sudo mv "/var/backups/static-pod-restart/${manifest}" /etc/kubernetes/manifests/
  sleep 25
done

# 不要一次移走全部控制平面 manifest
sudo crictl ps
```

**执行位置：`k8s-master` 宿主机；准备供控制端读取的临时副本。**

```bash
sudo install -m 0600 -o "$(id -u)" -g "$(id -g)" \
  /etc/kubernetes/admin.conf "$HOME/admin.conf.renewed"
```

**执行位置：Windows PowerShell；把续期后的配置从 Multipass 节点传到 Windows。**

```powershell
multipass transfer k8s-master:/home/ubuntu/admin.conf.renewed `
  "$env:USERPROFILE\.kube\admin.conf.renewed"
```

**执行位置：控制端 WSL；先备份现有配置，再替换并验证。**

```bash
cp ~/.kube/config ~/.kube/config.before-cert-renew
install -m 0600 \
  /mnt/c/Users/Administrator/.kube/admin.conf.renewed ~/.kube/config

kubectl get --raw='/readyz?verbose'
kubectl get nodes
kubectl get pods -n kube-system
```

**执行位置：`k8s-master` 宿主机；控制端复制成功后清理临时副本。**

```bash
rm -f "$HOME/admin.conf.renewed"
```

## 七、高可用（HA）控制平面

考试主要要求**理解概念**。

### 7.1 两种 etcd 拓扑

| 模式 | 说明 |
|------|------|
| **Stacked etcd** | etcd 与控制面组件同节点（kubeadm 默认） |
| **External etcd** | etcd 独立集群，隔离性更好 |

### 7.2 HA 要点

- 多个控制面节点 + 前面一个负载均衡器（LB）
- 初始化时指定 LB 地址：

> **高风险：HA 初始化仅可重建环境选做。** 不能在现有单控制面集群上再次执行 `kubeadm init`；以下只展示新建 HA 集群时的角色边界。

**执行位置：首个控制平面节点宿主机（全新、未初始化节点）。**

```bash
sudo kubeadm init --control-plane-endpoint=<LB-IP>:6443 --upload-certs
```

- 其他控制面节点加入：

**执行位置：其他待加入的控制平面节点宿主机，每台分别执行。**

```bash
sudo kubeadm join <LB-IP>:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash> \
  --control-plane \
  --certificate-key <key>
```

| 参数 | 含义 |
|------|------|
| `--control-plane-endpoint` | 稳定的 LB 入口（HA 必需） |
| `--upload-certs` | 上传证书供其他控制面共享 |
| `--control-plane` | 以控制面身份加入（而非 worker） |

## 八、Helm 与 Kustomize

### 8.1 Helm（Kubernetes 包管理器）

把一组资源打包成「Chart」，一条命令安装。

**执行位置：控制端 WSL；使用专用实验命名空间 `helm-lab`。**

```bash
kubectl create namespace helm-lab
helm repo add bitnami https://charts.bitnami.com/bitnami   # 加仓库
helm repo update                                           # 更新索引
helm install my-nginx bitnami/nginx -n helm-lab            # 安装
helm list -A                                               # 列出已安装 release
helm upgrade my-nginx bitnami/nginx -n helm-lab --set replicaCount=3
helm uninstall my-nginx -n helm-lab
```

| 命令 | 作用 |
|------|------|
| `repo add/update` | 管理 Chart 仓库 |
| `install <名> <chart>` | 安装一个 release |
| `--set key=value` | 覆盖 Chart 默认值 |
| `list -A` | 所有命名空间的 release |

**只渲染不安装（2026 高频题型）：**

考试可能要求「用 Helm **生成 manifest** 而不真正安装」，如指定版本、禁用 CRD：

**执行位置：控制端 WSL；只生成本地文件，不提交到集群。**

```bash
helm template my-app bitnami/nginx \
  --namespace helm-lab \
  --version 15.0.0 \
  --set replicaCount=2 > out.yaml       # 渲染成 YAML 保存，不部署

helm search repo nginx --versions        # 查可用的 chart 版本
helm show values bitnami/nginx           # 查看 chart 可配置项
```

| 参数 | 含义 |
|------|------|
| `helm template` | 本地渲染 Chart 为 YAML，**不连集群、不安装** |
| `--version` | 指定 Chart 版本（注意是 Chart 版本，不是应用版本） |
| `--set` / `-f values.yaml` | 覆盖默认值 |
| `--include-crds` / `--skip-crds` | 是否包含/跳过 CRD |

### 8.2 Kustomize（kubectl 内置）

不改原始 YAML，用「叠加（overlay）」方式定制。

**执行位置：控制端 WSL；使用专用实验命名空间 `kustomize-lab`。**

```bash
kubectl create namespace kustomize-lab
kubectl kustomize ./base
kubectl apply -k ./overlays/production -n kustomize-lab
```

| 参数 | 含义 |
|------|------|
| `-k 目录` | 使用该目录的 kustomization.yaml |
| `kubectl kustomize` | 只渲染不应用，用于预览 |

## 九、CRD 与 Operator

- **CRD（CustomResourceDefinition）**：扩展 Kubernetes API，注册**自定义资源类型**（比如 `IPPool`）。
- **Operator**：CRD + 自定义控制器，把「运维知识」代码化，自动管理复杂应用。
- 考试要求：**理解概念**，能识别/查看 CRD 资源，可能涉及用 Helm 装个 Operator。

**执行位置：控制端 WSL。**

```bash
kubectl get crd                       # 列出所有自定义资源类型
kubectl api-resources                 # 查所有资源类型（含 CRD）
kubectl get <custom-resource> -A      # 查看某自定义资源
```

## 十一、易错点与自检清单

### 易错点

- etcd 证书路径**从 etcd.yaml 复制**，别背。
- etcd 恢复时宿主机 `hostPath.path` 指向恢复目录，容器内 `mountPath` 与 `--data-dir` 仍保持 `/var/lib/etcd`。
- 升级：控制面 `apply`，工作节点 `node`；工作节点先 `drain`。
- 升级前后 `apt-mark unhold`/`hold` 别漏。
- RBAC：ClusterRole + RoleBinding = 只在该命名空间生效。
- 每道 RBAC 题用 `auth can-i` 验证。
