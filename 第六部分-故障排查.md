## 〇、本章考什么

| 常见题型 | 排查方向 |
|----------|----------|
| Pod 起不来（Pending/CrashLoop/ImagePull） | describe + logs |
| 节点 NotReady | kubelet / containerd |
| 控制面组件异常 | 静态 Pod manifest / 日志 |
| Service 访问不通 | Endpoints / selector / DNS |
| 权限被拒 | RBAC / auth can-i |

**心法：** 先 `get` 看全局，再 `describe` 看 Events，再 `logs` 看细节。**Events 是排错的第一现场。**

---

## 一、通用排查流程（先背下来）

遇到任何问题，先按这个顺序过一遍：

```
1. kubectl get nodes                          → 节点是否 Ready？
2. kubectl get pods -A                         → 哪些 Pod 异常？
3. kubectl describe <资源> <名字>              → 看 Events（最关键）
4. kubectl logs <pod> [--previous]             → 看容器日志
5. kubectl get events --sort-by='.lastTimestamp'  → 按时间看全局事件
```

| 命令 | 作用 |
|------|------|
| `get nodes` | 先确认节点层面是否健康 |
| `get pods -A` | `-A` = 所有命名空间，快速定位异常 Pod |
| `describe` | 底部 Events 往往直接写出原因 |
| `logs --previous` | 看**上次崩溃**的日志（容器已重启时用） |
| `get events --sort-by` | 按时间排序，看最近发生了什么 |

## 二、Pod 状态排查

### 2.1 状态对照表

| 状态 | 常见原因 | 排查方向 |
|------|----------|----------|
| **Pending** | 调度不上 | 资源不足、PVC 未绑定、nodeSelector/亲和/污点不匹配 |
| **ImagePullBackOff / ErrImagePull** | 拉镜像失败 | 镜像名/tag 错、私有仓库无凭证、网络（国内常见） |
| **CrashLoopBackOff** | 容器启动即崩溃 | 应用报错、命令写错、缺 ConfigMap/Secret |
| **CreateContainerConfigError** | 配置错误 | 引用的 ConfigMap/Secret 或 key 不存在 |
| **OOMKilled** | 内存超 limit | 调大 memory limit |
| **Error / Failed** | 运行出错 | describe + logs |

### 2.2 常用命令

```bash
kubectl get pod <pod> -o wide -n <ns>       # 状态、IP、节点
kubectl describe pod <pod> -n <ns>          # Events：调度/拉镜像/启动原因
kubectl logs <pod> -n <ns>                  # 当前容器日志
kubectl logs <pod> --previous -n <ns>       # 上次崩溃的日志
kubectl logs <pod> -c <container> -n <ns>   # 多容器指定容器
kubectl exec <pod> -n <ns> -- env           # 看环境变量是否注入
```

### 2.3 分状态处理

**Pending：**

```bash
kubectl describe pod <pod> -n <ns>    # 看 Events 里的调度失败原因
kubectl describe nodes | grep -A5 "Allocated resources"   # 资源够不够
kubectl get pvc -n <ns>               # PVC 是否 Bound
```

常见：节点资源不足、污点无容忍、nodeSelector 没匹配节点、PVC 未绑定。

**ImagePullBackOff：**

```bash
kubectl describe pod <pod> -n <ns> | grep -A3 -i "failed\|image"
```

看是镜像名错，还是网络超时（国内改国内镜像源，见第一部分）。

**CrashLoopBackOff：**

```bash
kubectl logs <pod> --previous -n <ns>   # 关键：看崩溃前输出
```

多为应用配置错、依赖缺失、启动命令错。

## 三、节点故障排查

### 3.1 从控制面看

```bash
kubectl get nodes                       # 谁 NotReady
kubectl describe node <node>            # Conditions + Events
```

`describe node` 里看 **Conditions**：

- `Ready=False` + `KubeletNotReady`：kubelet 或 CNI 问题
- `MemoryPressure` / `DiskPressure=True`：资源压力

### 3.2 到节点上排查（SSH / multipass shell）

```bash
sudo systemctl status kubelet                       # kubelet 是否在跑
sudo systemctl restart kubelet                      # 重启 kubelet
sudo journalctl -u kubelet --no-pager | tail -50    # kubelet 日志
sudo journalctl -u containerd --no-pager | tail -50 # 运行时日志
```

| 命令 | 作用 |
|------|------|
| `systemctl status kubelet` | 服务是否 active |
| `journalctl -u kubelet` | kubelet 日志（连不上 API、cgroup、证书等） |
| `journalctl -u containerd` | 运行时日志 |

### 3.3 NotReady 常见原因

| 原因 | 检查/修复 |
|------|-----------|
| kubelet 停了 | `systemctl start kubelet` |
| swap 被打开 | `swapoff -a` |
| 容器运行时挂了 | `systemctl status containerd` |
| CNI 未就绪 | 看 calico-node Pod；`cni plugin not initialized` |
| 证书过期 | `kubeadm certs check-expiration` |
| 磁盘满 | `df -h`，清理空间 |

## 四、控制平面组件故障

控制面组件是**静态 Pod**（第二/五部分讲过），manifest 在 `/etc/kubernetes/manifests/`。

```bash
kubectl get pods -n kube-system                 # 各组件是否 Running
ls /etc/kubernetes/manifests/                   # etcd/apiserver/scheduler/cm

# 组件日志
kubectl logs -n kube-system kube-apiserver-<node>
kubectl logs -n kube-system etcd-<node>
kubectl logs -n kube-system kube-scheduler-<node>
kubectl logs -n kube-system kube-controller-manager-<node>
```

### 4.1 apiserver 起不来（kubectl 都连不上时）

apiserver 挂了 `kubectl` 会失败，此时到 master 上用容器运行时看：

```bash
sudo crictl ps -a | grep apiserver           # 容器状态
sudo crictl logs <container-id>              # 容器日志
sudo journalctl -u kubelet -e | tail -50     # kubelet 起静态 Pod 的日志
```

常见：改错了 `kube-apiserver.yaml`（参数/缩进），kubelet 起不来该 Pod。改回 manifest 即可，kubelet 会自动重建。

### 4.2 常见修复对照

| 问题 | 修复 |
|------|------|
| kubelet 未运行 | `systemctl start kubelet` |
| manifest YAML 写错 | 修 `/etc/kubernetes/manifests/` 下文件（注意缩进） |
| 证书过期 | `kubeadm certs renew all` + 重启组件 |
| etcd data-dir 错 | 修 `etcd.yaml` 的 `--data-dir` |

### 4.3 etcd 健康

```bash
ETCDCTL_API=3 etcdctl endpoint health \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
```

## 五、网络故障排查

「Service 访问不通」是高频题。按这条链逐段查：

```bash
# ① Service 存不存在、类型对不对
kubectl get svc <svc> -n <ns>

# ② Endpoints 有没有后端 Pod（空 = selector/label 不匹配）
kubectl get endpoints <svc> -n <ns>

# ③ selector 和 Pod label 对不对
kubectl get svc <svc> -n <ns> -o yaml | grep -A3 selector
kubectl get pods -n <ns> --show-labels

# ④ Pod 端口是否真的在监听（targetPort 对不对）
kubectl exec <pod> -n <ns> -- wget -qO- localhost:<port>

# ⑤ DNS 是否正常
kubectl run test --image=busybox:1.36 --rm -it -- nslookup <svc>.<ns>

# ⑥ kube-proxy / CoreDNS 是否正常
kubectl get pods -n kube-system -l k8s-app=kube-proxy
kubectl get pods -n kube-system -l k8s-app=kube-dns

# ⑦ 有没有 NetworkPolicy 挡住
kubectl get networkpolicy -n <ns>
```

### 5.1 最常见的 5 个网络问题

1. **Service selector 与 Pod label 不匹配**（拼写错）→ Endpoints 为空
2. **targetPort 与容器端口不一致** → 通但无响应
3. **NetworkPolicy 挡了流量且没放行 DNS**
4. **CoreDNS Pod 没运行** → 所有域名解析失败
5. **kube-proxy 异常** → Service 转发不生效

> 口诀：**不通先看 Endpoints，空就查 label；有 Endpoints 还不通，查 targetPort 和 NetworkPolicy。**

## 六、存储故障排查

```bash
kubectl get pvc -n <ns>                  # STATUS 是否 Bound
kubectl describe pvc <pvc> -n <ns>       # Events 里看 Pending 原因
kubectl get pv                           # PV 状态
```

PVC Pending 三查（见第四部分）：容量、accessModes、storageClassName。

## 七、RBAC 权限故障

「操作被拒绝（Forbidden）」类问题：

```bash
# 检查某主体能否做某操作
kubectl auth can-i <verb> <resource> --as=<user> -n <ns>
kubectl auth can-i list pods -n dev --as=system:serviceaccount:dev:my-sa

# 看绑定关系
kubectl get rolebinding,clusterrolebinding -A | grep <主体>
kubectl describe rolebinding <name> -n <ns>
```

常见：Role 少了 verb/resource，或 Binding 绑错命名空间/主体。修 Role 或 RoleBinding 后再 `auth can-i` 验证。

## 八、日志位置汇总

| 组件 | 日志查看方式 |
|------|-------------|
| kubelet | `journalctl -u kubelet` |
| containerd | `journalctl -u containerd` |
| kube-apiserver | `kubectl logs -n kube-system kube-apiserver-<node>` |
| etcd | `kubectl logs -n kube-system etcd-<node>` |
| 应用容器 | `kubectl logs <pod>` / `--previous` |
| apiserver 挂了时 | `crictl ps -a` + `crictl logs <id>` |

> 记住分层：**节点服务（kubelet/containerd）用 journalctl；集群内 Pod（含控制面静态 Pod）用 kubectl logs；apiserver 都挂了用 crictl。**

## 九、资源监控

```bash
kubectl top nodes                          # 节点 CPU/内存（需 metrics-server）
kubectl top pods -n <ns>                    # Pod 资源用量
kubectl describe node <node>               # 看 Allocated resources
kubectl get events --sort-by='.lastTimestamp' -A   # 全局事件按时间
```

> `top` 需要集群装了 **metrics-server**；没装会报错，属正常。

## 十、综合演练场景

用你现有集群故意「制造故障」再修复，是最好的练习。

### 场景 1：Service 访问不通

```bash
# 制造：把 Service selector 改错
kubectl patch svc web -n dev -p '{"spec":{"selector":{"app":"wrong"}}}'
curl http://10.13.31.10:31631        # 不通了
kubectl get endpoints web -n dev     # 空！

# 修复：改回正确 selector
kubectl patch svc web -n dev -p '{"spec":{"selector":{"app":"web"}}}'
kubectl get endpoints web -n dev     # 恢复
```

### 场景 2：Pod CrashLoop

```bash
# 制造：用错误命令跑 Pod
kubectl run bad --image=busybox -n dev -- /bin/false
kubectl get pod bad -n dev           # CrashLoopBackOff
kubectl describe pod bad -n dev      # 看 Events
kubectl logs bad --previous -n dev   # 看崩溃日志
kubectl delete pod bad -n dev        # 清理
```

### 场景 3：节点 NotReady

```bash
# 在某 worker 上停 kubelet
sudo systemctl stop kubelet
kubectl get nodes                    # 该节点 NotReady
# 修复
sudo systemctl start kubelet
kubectl get nodes                    # 恢复 Ready
```

> 练习时注意：别在临考前把集群玩坏了修不回来。建议在专门的练习命名空间/可重建环境操作。

## 十二、易错点与自检清单

### 易错点

- CrashLoop 要看 `logs --previous`，不是当前 logs。
- apiserver 挂了 kubectl 用不了，改用 `crictl`。
- 静态 Pod 改坏了改回 manifest 即可，别 `kubectl delete`。
- Service 不通先看 **Endpoints**。
- `top` 报错多半是没装 metrics-server，不是故障。

### 自检清单

- [ ] 能背出通用排查 5 步流程
- [ ] 能处理 Pending / ImagePullBackOff / CrashLoopBackOff
- [ ] 能排查节点 NotReady（kubelet/containerd）
- [ ] 能排查控制面静态 Pod（含 apiserver 挂掉时用 crictl）
- [ ] 能沿 Service→Endpoints→label→targetPort→DNS 链排网络
- [ ] 能用 `auth can-i` 排 RBAC 权限
- [ ] 知道各组件日志在哪看
