# 第六部分：故障排查（考纲 30%）

> 上一部分：[第五部分-集群架构安装与配置.md](./第五部分-集群架构安装与配置.md)　总纲：[CKA备考学习路线总纲.md](./CKA备考学习路线总纲.md)  
> 这是**分值最高**的域（30%），而且综合了前面所有知识。关键不是记命令，而是建立**固定排查流程**，避免随机试错浪费时间。默认零基础，逐命令、逐参数讲解。

> **统一环境基线**
>
> - 控制端：WSL，登录提示符 `Administrator@DESKTOP-M9MJEIU`，用于执行 `kubectl` 和保存取证结果；进入 Multipass 节点时，从 Windows PowerShell 使用 `multipass shell <节点名>`。
> - 控制平面：`k8s-master`，集群网卡 `eth1`，固定 IP `10.13.31.10`。
> - 工作节点：`k8s-worker1=10.13.31.11`、`k8s-worker2=10.13.31.12`，集群网卡均为 `eth1`。
> - 容器运行时：containerd；CNI：Calico，封装模式为 IPIP。
> - 除非命令块另有说明，`kubectl` 在**控制端 WSL**执行；`systemctl`、`journalctl`、`ip`、`crictl` 在标明的**目标节点宿主机**执行；`kubectl exec ... --` 后面的命令才在**目标容器内**执行。
> - 排障原则：**先观察和保存日志，再做重启或修改**。重启可能清掉关键现场，不能作为第一步。

## 〇、本章考什么

| 常见题型 | 排查方向 |
|----------|----------|
| Pod 起不来（Pending/CrashLoop/ImagePull） | describe + logs |
| 节点 NotReady | kubelet / containerd |
| 控制面组件异常 | 静态 Pod manifest / 日志 |
| Service 访问不通 | Endpoints / selector / DNS |
| 权限被拒 | RBAC / auth can-i |

**心法：** 先 `get` 看全局，再 `describe` 看 Events，再 `logs` 看细节。**Events 是排错的第一现场。**

## 一、通用排查流程（先背下来）

遇到任何问题，按“控制端观察 → 目标节点取证 → 目标容器内验证”的顺序逐层缩小范围：

**执行位置：控制端 WSL。**

```
1. kubectl get nodes -o wide                        → 节点是否 Ready，INTERNAL-IP 是否正确？
2. kubectl get pods -A -o wide                      → 哪些 Pod 异常、落在哪个节点？
3. kubectl describe <资源> <名字>                   → 看 Events（最关键）
4. kubectl logs <pod> [--previous]                  → 先保存当前/上次容器日志
5. kubectl get events -A --sort-by='.lastTimestamp' → 按时间看全局事件
```

若控制端已定位到具体节点，再通过 Multipass 进入该节点检查 kubelet、containerd、CNI 与路由；只有 Pod 仍在运行且需要验证进程视角时，才进入目标容器。**先日志，后重启；每次只改一个变量。**

| 命令 | 作用 |
|------|------|
| `get nodes` | 先确认节点层面是否健康 |
| `get pods -A` | `-A` = 所有命名空间，快速定位异常 Pod |
| `describe` | 底部 Events 往往直接写出原因 |
| `logs --previous` | 看**上次崩溃**的日志（容器已重启时用） |
| `get events --sort-by` | 按时间排序，看最近发生了什么 |

## 二、Pod 状态排查

### 2.1 状态对照表

| 状态 | 常见原因 | 排查方向 |
|------|----------|----------|
| **Pending** | 调度不上 | 资源不足、PVC 未绑定、nodeSelector/亲和/污点不匹配 |
| **ImagePullBackOff / ErrImagePull** | 拉镜像失败 | 镜像名/tag 错、私有仓库无凭证、网络（国内常见） |
| **CrashLoopBackOff** | 容器启动即崩溃 | 应用报错、命令写错、缺 ConfigMap/Secret |
| **CreateContainerConfigError** | 配置错误 | 引用的 ConfigMap/Secret 或 key 不存在 |
| **OOMKilled** | 内存超 limit | 调大 memory limit |
| **Error / Failed** | 运行出错 | describe + logs |

### 2.2 常用命令

**执行位置：控制端 WSL；最后一条通过 API 进入目标容器。**

```bash
kubectl get pod <pod> -o wide -n <ns>       # 状态、IP、节点
kubectl describe pod <pod> -n <ns>          # Events：调度/拉镜像/启动原因
kubectl logs <pod> -n <ns>                  # 当前容器日志
kubectl logs <pod> --previous -n <ns>       # 上次崩溃的日志
kubectl logs <pod> -c <container> -n <ns>   # 多容器指定容器
kubectl exec <pod> -n <ns> -- env           # 看环境变量是否注入
```

### 2.3 分状态处理

**Pending：**

**执行位置：控制端 WSL。**

```bash
kubectl describe pod <pod> -n <ns>    # 看 Events 里的调度失败原因
kubectl describe nodes | grep -A5 "Allocated resources"   # 资源够不够
kubectl get pvc -n <ns>               # PVC 是否 Bound
```

常见：节点资源不足、污点无容忍、nodeSelector 没匹配节点、PVC 未绑定。

**ImagePullBackOff：**

**执行位置：控制端 WSL。**

```bash
kubectl describe pod <pod> -n <ns> | grep -A3 -i "failed\|image"
```

看是镜像名错，还是网络超时（国内改国内镜像源，见第一部分）。

**CrashLoopBackOff：**

**执行位置：控制端 WSL。**

```bash
kubectl logs <pod> --previous -n <ns>   # 关键：看崩溃前输出
```

多为应用配置错、依赖缺失、启动命令错。

## 三、节点故障排查

### 3.1 控制端观察

**执行位置：控制端 WSL。**

```bash
kubectl get nodes -o wide               # 谁 NotReady，INTERNAL-IP 是否为 eth1 地址
kubectl describe node <node>            # Conditions + Events
kubectl get pods -A -o wide --field-selector spec.nodeName=<node>
kubectl get events -A --sort-by='.lastTimestamp'
```

`describe node` 里看 **Conditions**：

- `Ready=False` + `KubeletNotReady`：kubelet 或 CNI 问题
- `MemoryPressure` / `DiskPressure=True`：资源压力

### 3.2 目标节点取证

先取证，不要一上来就 `restart kubelet`。先从 Windows PowerShell 执行 `multipass shell <节点名>` 进入目标节点，再执行以下命令。

**执行位置：发生异常的目标节点宿主机。**

```bash
sudo systemctl status kubelet --no-pager
sudo systemctl status containerd --no-pager
sudo journalctl -u kubelet --since "-20 min" --no-pager
sudo journalctl -u containerd --since "-20 min" --no-pager
sudo crictl ps -a
df -h
free -h
swapon --show
```

| 命令 | 作用 |
|------|------|
| `systemctl status kubelet` | 服务是否 active |
| `journalctl -u kubelet` | kubelet 日志（连不上 API、cgroup、证书等） |
| `journalctl -u containerd` | 运行时日志 |

确认日志指向服务卡死，且已保存现场后，才重启对应服务。

**执行位置：发生异常的目标节点宿主机；仅在证据支持时执行。**

```bash
sudo systemctl restart containerd
sudo systemctl restart kubelet
sudo systemctl status containerd kubelet --no-pager
```

**执行位置：控制端 WSL；修复后验证。**

```bash
kubectl get nodes -o wide
kubectl get pods -A -o wide
kubectl get events -A --sort-by='.lastTimestamp'
```

### 3.3 目标容器内验证

容器仍在运行但应用行为异常时，再从容器视角验证配置、监听端口和 DNS。CrashLoop 容器通常无法稳定进入，应优先看 `logs --previous`。

**执行位置：控制端 WSL 发起；`--` 后命令在目标容器内执行。**

```bash
kubectl exec -n <ns> <pod> -c <container> -- env
kubectl exec -n <ns> <pod> -c <container> -- cat /etc/resolv.conf
kubectl exec -n <ns> <pod> -c <container> -- sh -c 'ss -lnt || netstat -lnt'
```

### 3.4 NotReady 常见原因

| 原因 | 检查/修复 |
|------|-----------|
| kubelet 停了 | `systemctl start kubelet` |
| swap 被打开 | `swapoff -a` |
| 容器运行时挂了 | `systemctl status containerd` |
| CNI 未就绪 | 看 calico-node Pod；`cni plugin not initialized` |
| 证书过期 | `kubeadm certs check-expiration` |
| 磁盘满 | `df -h`，清理空间 |

## 四、控制平面组件故障

控制面组件是**静态 Pod**（第二/五部分讲过），manifest 在 `/etc/kubernetes/manifests/`。

先从控制端观察；不要把控制端的 `kubectl` 与 master 宿主机的文件命令混在一个命令块里。

**执行位置：控制端 WSL。**

```bash
kubectl get pods -n kube-system                 # 各组件是否 Running
kubectl logs -n kube-system kube-apiserver-<node>
kubectl logs -n kube-system etcd-<node>
kubectl logs -n kube-system kube-scheduler-<node>
kubectl logs -n kube-system kube-controller-manager-<node>
```

**执行位置：`k8s-master` 宿主机。**

```bash
sudo ls -l /etc/kubernetes/manifests/
sudo journalctl -u kubelet --since "-20 min" --no-pager
sudo crictl ps -a
```

### 4.1 apiserver 起不来（kubectl 都连不上时）

apiserver 挂了 `kubectl` 会失败，此时到 master 上用容器运行时看：

**执行位置：`k8s-master` 宿主机。**

```bash
sudo crictl ps -a --name kube-apiserver
sudo crictl logs <container-id>
sudo journalctl -u kubelet --since "-20 min" --no-pager
```

常见：改错了 `kube-apiserver.yaml`（参数/缩进），kubelet 起不来该 Pod。改回 manifest 即可，kubelet 会自动重建。

### 4.2 常见修复对照

| 问题 | 修复 |
|------|------|
| kubelet 未运行 | `systemctl start kubelet` |
| manifest YAML 写错 | 修 `/etc/kubernetes/manifests/` 下文件（注意缩进） |
| 证书过期 | `kubeadm certs renew all` + 重启组件 |
| etcd data-dir 错 | 修 `etcd.yaml` 的 `--data-dir` |

### 4.3 etcd 健康

**执行位置：`k8s-master` 宿主机。**

```bash
sudo ETCDCTL_API=3 etcdctl endpoint health \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
  --key=/etc/kubernetes/pki/etcd/healthcheck-client.key
```

## 五、网络故障排查

「Service 访问不通」是高频题。按这条链逐段查：

**执行位置：控制端 WSL。**

```bash
# ① Service 存不存在、类型对不对
kubectl get svc <svc> -n <ns>

# ② Endpoints 有没有后端 Pod（空 = selector/label 不匹配）
kubectl get endpoints <svc> -n <ns>

# ③ selector 和 Pod label 对不对
kubectl get svc <svc> -n <ns> -o yaml | grep -A3 selector
kubectl get pods -n <ns> --show-labels

# ④ kube-proxy / CoreDNS 是否正常
kubectl get pods -n kube-system -l k8s-app=kube-proxy
kubectl get pods -n kube-system -l k8s-app=kube-dns

# ⑤ 有没有 NetworkPolicy 挡住
kubectl get networkpolicy -n <ns>
```

**执行位置：控制端 WSL 发起；`--` 后命令在目标容器内执行。**

```bash
# ⑥ 容器实际监听端口是否与 targetPort 一致
kubectl exec <pod> -n <ns> -- wget -qO- localhost:<port>

# ⑦ 从临时容器验证 Service DNS
kubectl run dns-test -n <ns> --image=busybox:1.36 --rm -it \
  --restart=Never -- nslookup <svc>.<ns>.svc.cluster.local
```

### 5.1 双网卡与 INTERNAL-IP

本环境的集群通信必须走 `eth1`：`k8s-master=10.13.31.10`、`k8s-worker1=.11`、`k8s-worker2=.12`。双网卡机器如果把默认出口网卡地址注册成 `INTERNAL-IP`，Calico 也可能选错源地址，表现为节点 Ready 但跨节点 Pod 网络不通。

**执行位置：控制端 WSL。**

```bash
kubectl get nodes -o wide
kubectl get node k8s-master -o jsonpath='{.status.addresses}'; echo
kubectl get node k8s-worker1 -o jsonpath='{.status.addresses}'; echo
kubectl get node k8s-worker2 -o jsonpath='{.status.addresses}'; echo
```

预期 `INTERNAL-IP` 分别为 `10.13.31.10/11/12`。若不是，检查 kubelet 的 `--node-ip` 来源；先取证，修改前备份配置。

**执行位置：INTERNAL-IP 异常的目标节点宿主机。**

```bash
ip -br address
ip route
ip route get 10.13.31.10
sudo grep -R -- "--node-ip" /var/lib/kubelet /etc/default/kubelet /etc/systemd/system/kubelet.service.d
sudo journalctl -u kubelet --since "-20 min" --no-pager
```

### 5.2 Calico 地址自动探测

检查 calico-node 是否明确按 `eth1` 探测地址。常见合理值是 `interface=eth1`；也可能使用 `kubernetes-internal-ip`，此时必须先保证节点 INTERNAL-IP 正确。

**执行位置：控制端 WSL。**

```bash
kubectl get ds calico-node -n kube-system -o yaml | \
  grep -A4 -E 'IP_AUTODETECTION_METHOD|IP6_AUTODETECTION_METHOD'
kubectl get ippools.crd.projectcalico.org -o yaml | \
  grep -E 'name:|cidr:|ipipMode:|natOutgoing:'
kubectl get pods -n kube-system -l k8s-app=calico-node -o wide
kubectl logs -n kube-system <calico-node-pod> -c calico-node --since=20m
```

不要在未确认现有安装方式前直接覆盖 DaemonSet；Operator、Helm 和原始 manifest 管理的配置入口可能不同。

### 5.3 IPIP、tunl0 与路由

Calico IPIP 跨节点流量通常经过 `tunl0`。依次确认 `eth1` 到对端节点可达、`tunl0` 存在且为 UP、Pod CIDR 路由指向正确隧道/下一跳。

**执行位置：出现跨节点不通的目标节点宿主机。**

```bash
ip -br link show eth1
ip -br address show eth1
ping -c 3 10.13.31.10
ip -d link show tunl0
ip address show tunl0
ip route | grep -E 'tunl0|blackhole|proto bird'
sudo journalctl -u kubelet --since "-20 min" --no-pager | grep -iE 'cni|calico|network'
```

**执行位置：控制端 WSL 发起；`--` 后命令在目标 calico-node 容器内执行。**

```bash
kubectl exec -n kube-system <calico-node-pod> -c calico-node -- \
  calico-node -bird-ready
kubectl exec -n kube-system <calico-node-pod> -c calico-node -- \
  ip route
```

若 `tunl0` 缺失或路由不对，先对比三台节点的 calico-node 日志、自动探测地址和 IPPool 的 `ipipMode`，不要先重启全部 calico-node。

### 5.4 最常见的网络问题

1. **Service selector 与 Pod label 不匹配**（拼写错）→ Endpoints 为空
2. **targetPort 与容器端口不一致** → 通但无响应
3. **NetworkPolicy 挡了流量且没放行 DNS**
4. **CoreDNS Pod 没运行** → 所有域名解析失败
5. **kube-proxy 异常** → Service 转发不生效
6. **双网卡选错 INTERNAL-IP / Calico autodetection 地址** → 跨节点网络异常
7. **IPIP 的 `tunl0` 或 Pod CIDR 路由异常** → 同节点通、跨节点不通

> 口诀：**不通先看 Endpoints，空就查 label；有 Endpoints 还不通，查 targetPort 和 NetworkPolicy。**

## 七、RBAC 权限故障

「操作被拒绝（Forbidden）」类问题：

**执行位置：控制端 WSL。**

```bash
# 检查某主体能否做某操作
kubectl auth can-i <verb> <resource> --as=<user> -n <ns>
kubectl auth can-i list pods -n dev --as=system:serviceaccount:dev:my-sa

# 看绑定关系
kubectl get rolebinding,clusterrolebinding -A | grep <主体>
kubectl describe rolebinding <name> -n <ns>
```

常见：Role 少了 verb/resource，或 Binding 绑错命名空间/主体。修 Role 或 RoleBinding 后再 `auth can-i` 验证。

## 八、日志位置汇总

| 组件 | 日志查看方式 |
|------|-------------|
| kubelet | `journalctl -u kubelet` |
| containerd | `journalctl -u containerd` |
| kube-apiserver | `kubectl logs -n kube-system kube-apiserver-<node>` |
| etcd | `kubectl logs -n kube-system etcd-<node>` |
| 应用容器 | `kubectl logs <pod>` / `--previous` |
| apiserver 挂了时 | `crictl ps -a` + `crictl logs <id>` |

> 记住分层：**节点服务（kubelet/containerd）用 journalctl；集群内 Pod（含控制面静态 Pod）用 kubectl logs；apiserver 都挂了用 crictl。**

## 九、资源监控

**执行位置：控制端 WSL。**

```bash
kubectl top nodes                          # 节点 CPU/内存（需 metrics-server）
```

## 十、综合演练场景

所有命名空间级故障注入统一放在 `troubleshooting-lab`，不要污染 `dev` 或业务命名空间。

**执行位置：控制端 WSL。**

```bash
kubectl create namespace troubleshooting-lab \
  --dry-run=client -o yaml | kubectl apply -f -
```

### 场景 1：Service 访问不通

先创建独立实验资源，并把原 selector 保存到本地文件和 shell 变量，之后再注入错误。

**执行位置：控制端 WSL。**

```bash
kubectl create deployment web -n troubleshooting-lab --image=nginx:1.27
kubectl expose deployment web -n troubleshooting-lab --port=80
kubectl rollout status deployment/web -n troubleshooting-lab

kubectl get svc web -n troubleshooting-lab -o json \
  > /tmp/troubleshooting-lab-web-service.before.json
ORIGINAL_SELECTOR=$(kubectl get svc web -n troubleshooting-lab \
  -o jsonpath='{.spec.selector.app}')
test -n "$ORIGINAL_SELECTOR"

kubectl patch svc web -n troubleshooting-lab \
  -p '{"spec":{"selector":{"app":"wrong"}}}'
kubectl get endpointslice -n troubleshooting-lab \
  -l kubernetes.io/service-name=web

# 无论观察结果是否符合预期，都执行恢复和健康验证
kubectl patch svc web -n troubleshooting-lab \
  -p "{\"spec\":{\"selector\":{\"app\":\"${ORIGINAL_SELECTOR}\"}}}"
kubectl get svc,endpointslice,pods -n troubleshooting-lab -o wide
kubectl run service-check -n troubleshooting-lab --image=busybox:1.36 \
  --restart=Never --rm -it -- wget -qO- http://web
```

### 场景 2：Pod CrashLoop

**执行位置：控制端 WSL。**

```bash
# 制造：用错误命令跑 Pod
kubectl run bad --image=busybox:1.36 -n troubleshooting-lab -- /bin/false
kubectl get pod bad -n troubleshooting-lab
kubectl describe pod bad -n troubleshooting-lab
kubectl logs bad --previous -n troubleshooting-lab

# 无条件清理并确认实验 Pod 已删除
kubectl delete pod bad -n troubleshooting-lab --ignore-not-found
kubectl get pods -n troubleshooting-lab
```

### 场景 3：节点 NotReady

> 此注入**只允许在 `k8s-worker2`** 操作。kubelet 停止后节点不会瞬间 NotReady；控制器需等待心跳/租约超时，通常要几十秒。先注册 5 分钟自动恢复任务，确保会无条件拉起 kubelet。

**执行位置：`k8s-worker2`（`10.13.31.12`）宿主机。**

```bash
# 安全网：即使节点终端断开，5 分钟后也会自动启动 kubelet
sudo systemd-run --unit=kubelet-lab-recovery --on-active=5m \
  /bin/systemctl start kubelet
sudo systemctl stop kubelet
```

**执行位置：控制端 WSL；等待并观察状态变化。**

```bash
kubectl get node k8s-worker2 -w
```

看到 `NotReady` 后按 `Ctrl+C` 退出观察。即使未在预期时间看到 NotReady，也必须执行下面恢复步骤。

**执行位置：`k8s-worker2` 宿主机；无条件执行恢复。**

```bash
sudo systemctl start kubelet
sudo systemctl stop kubelet-lab-recovery.timer 2>/dev/null || true
sudo systemctl status kubelet --no-pager
```

**执行位置：控制端 WSL；无条件执行健康验证。**

```bash
kubectl wait --for=condition=Ready node/k8s-worker2 --timeout=180s
kubectl get nodes -o wide
kubectl get pods -A -o wide --field-selector spec.nodeName=k8s-worker2
kubectl get events -A --sort-by='.lastTimestamp'
```

> 练习结束可删除命名空间：`kubectl delete namespace troubleshooting-lab`。节点级注入不受命名空间隔离，因此必须严格限定 `k8s-worker2` 并保留自动恢复安全网。

## 十二、易错点与自检清单

### 易错点

- CrashLoop 要看 `logs --previous`，不是当前 logs。
- apiserver 挂了 kubectl 用不了，改用 `crictl`。
- 静态 Pod 改坏了改回 manifest 即可，别 `kubectl delete`。
- Service 不通先看 **Endpoints**。
- `top` 报错多半是没装 metrics-server，不是故障。

- [ ] 能背出通用排查 5 步流程
