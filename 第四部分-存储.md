## 一、为什么需要持久化存储

容器里的文件默认是**临时的**：容器删了/重启了，写在容器文件系统里的数据就没了。

要让数据**长期保存**（数据库、上传文件等），需要把外部存储「挂」进容器。Kubernetes 用一套抽象来管理：**PV / PVC / StorageClass**。

核心思路：**把「提供存储」和「使用存储」解耦**。

- 管理员提供存储资源（PV / StorageClass）
- 用户只管申请（PVC），不用关心底层是什么盘

---

## 二、三大核心对象：PV / PVC / StorageClass

| 对象 | 全称 | 作用域 | 通俗理解 |
|------|------|--------|----------|
| **PV** | PersistentVolume | 集群级 | 一块**已经存在**的存储（管理员准备好的「房子」） |
| **PVC** | PersistentVolumeClaim | 命名空间级 | 用户对存储的**申请**（「我要一间 2G 的房子」） |
| **StorageClass** | StorageClass | 集群级 | **动态造 PV** 的模板（「按需盖房子的包工头」） |

关系：

```
静态：管理员建 PV  ←绑定→  用户建 PVC  ←挂载→  Pod
动态：用户建 PVC → StorageClass 自动造 PV → 绑定 → Pod
```

### 2.1 绑定条件（PVC 找 PV，三者必须同时满足）

1. PVC 请求的 `storage` **≤** PV 的 `capacity`（容量够）
2. `accessModes` **兼容**（访问模式匹配）
3. `storageClassName` **完全一致**（含都为空字符串的情况）

> **PVC 一直 Pending** 基本就是这三条里有一条不满足。

---

## 三、访问模式 Access Modes

描述存储能被怎样挂载。

| 模式 | 缩写 | 含义 |
|------|------|------|
| ReadWriteOnce | RWO | 被**单个节点**读写（最常用） |
| ReadOnlyMany | ROX | 被**多个节点**只读 |
| ReadWriteMany | RWX | 被**多个节点**读写（hostPath 不支持） |
| ReadWriteOncePod | RWOP | 被**单个 Pod**读写（v1.29+） |

> 考试里 `hostPath` / 本地盘一般只支持 **RWO**。要 RWX 需要 NFS 等网络存储。

---

## 四、回收策略 Reclaim Policy

删除 PVC 后，PV 怎么处理。

| 策略 | 行为 |
|------|------|
| **Retain** | PV **保留**，数据不删，需管理员手动清理（安全，考试常用） |
| **Delete** | PV 及底层存储**自动删除**（云 StorageClass 默认） |
| ~~Recycle~~ | 已废弃，不用管 |

在 PV 里配置：

```yaml
spec:
  persistentVolumeReclaimPolicy: Retain
```

## 五、Volume Mode

| **Filesystem** | 挂载为一个目录（默认，99% 的情况） |
| **Block** | 挂载为原始块设备（特殊场景） |

## 六、静态供应 vs 动态供应

### 6.1 静态供应（手动建 PV）

管理员先建 PV，用户再建 PVC 去绑定。

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  capacity:
    storage: 5Gi                       # 这块 PV 有 5G
  accessModes:
  - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: manual             # 自定义一个类名
  hostPath:                            # 用节点本地目录（考试常用，生产不用）
    path: /data/my-pv
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
  namespace: dev
spec:
  accessModes:
  - ReadWriteOnce                      # 要和 PV 兼容
  resources:
    requests:
      storage: 2Gi                     # 申请 2G（≤ PV 的 5G）
  storageClassName: manual             # 要和 PV 完全一致
```

创建后检查绑定：

```bash
kubectl get pv
kubectl get pvc -n dev
# PVC 的 STATUS 变成 Bound 表示绑定成功
```

| PV 字段 | 含义 |
|---------|------|
| `capacity.storage` | 提供的容量 |
| `accessModes` | 支持的访问模式 |
| `persistentVolumeReclaimPolicy` | 回收策略 |
| `storageClassName` | 类名，PVC 要对上 |
| `hostPath.path` | 节点上的目录 |

### 6.2 动态供应（StorageClass 自动建 PV）

用户建 PVC 时指定 StorageClass，系统自动造 PV，不用管理员手动建。

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast
provisioner: kubernetes.io/no-provisioner   # 谁来造 PV（本地存储示例）
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer     # 绑定时机
```

| 字段 | 含义 |
|------|------|
| `provisioner` | 存储供应器；云环境是对应插件，本地测试常用 `no-provisioner` |
| `reclaimPolicy` | 动态建出来的 PV 的回收策略 |
| `volumeBindingMode` | 见下表 |

**volumeBindingMode：**

| 模式 | 说明 |
|------|------|
| **Immediate** | PVC 一创建就立即绑定/创建 PV |
| **WaitForFirstConsumer** | 等到有 Pod 用它、调度确定节点后再绑定（避免 PV 建在错误节点） |

PVC 用动态供应：

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: dyn-pvc
  namespace: dev
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
  storageClassName: fast     # 指向 StorageClass，触发动态供应
```

查看默认 StorageClass：

```bash
kubectl get storageclass
# 名字后带 (default) 的是默认 SC；PVC 不写 storageClassName 时用它
```

## 七、Pod 挂载 PVC

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app
  namespace: dev
spec:
  containers:
  - name: app
    image: nginx:1.27
    volumeMounts:
    - name: data                       # 对应下面 volumes 的名字
      mountPath: /usr/share/nginx/html # 挂到容器里的路径
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-pvc                # 用哪个 PVC
```

| 字段 | 含义 |
|------|------|
| `volumeMounts.name` | 引用 `volumes` 里定义的卷 |
| `volumeMounts.mountPath` | 挂到容器内哪个目录 |
| `volumes.persistentVolumeClaim.claimName` | 使用的 PVC 名 |

验证：

```bash
kubectl exec app -n dev -- df -h | grep nginx
kubectl exec app -n dev -- ls /usr/share/nginx/html
```

## 八、其他常用 Volume 类型

除了 PVC，Pod 还能直接挂这些卷：

| 类型 | 用途 | 生命周期 |
|------|------|----------|
| **emptyDir** | Pod 内容器间共享的临时目录 | 随 Pod 删除而消失 |
| **hostPath** | 挂节点上的目录 | 与节点绑定（考试练习常用） |
| **configMap** | 把 ConfigMap 变成文件 | 见第二部分 |
| **secret** | 把 Secret 变成文件 | 见第二部分 |

```yaml
  volumes:
  - name: cache
    emptyDir: {}                # 临时空目录
  - name: config-vol
    configMap:
      name: app-config
  - name: secret-vol
    secret:
      secretName: db-creds
```

## 九、存储故障排查

```bash
kubectl get pvc -n dev                    # STATUS 是否 Bound
kubectl describe pvc my-pvc -n dev        # Events 里看为什么 Pending
kubectl get pv                            # PV 状态：Available/Bound/Released
kubectl describe pv my-pv
```

**PVC 一直 Pending 的排查顺序：**

1. `storageClassName` 是否和 PV / 存在的 SC 一致？
2. `accessModes` 是否兼容？
3. 申请容量是否 ≤ PV 容量？
4. 是否根本没有匹配的 PV，且没有可用的 StorageClass 做动态供应？

| PV 状态 | 含义 |
|---------|------|
| `Available` | 空闲，未绑定 |
| `Bound` | 已绑定到某 PVC |
| `Released` | PVC 已删但 PV（Retain）还未回收 |

## 十一、易错点与自检清单

### 易错点

- **绑定三条件**：容量、accessModes、storageClassName，缺一不绑。
- `storageClassName: ""`（空串）只匹配**同样没有 SC** 的 PV，不等于「不写」。
- `hostPath` 不支持 RWX。
- 删除顺序：**先删 Pod → 再删 PVC → 最后处理 PV**。
- Retain 的 PV 在 PVC 删除后进入 `Released`，不会自动可用，需手动处理。
