## 一、为什么需要持久化存储

容器里的文件默认是**临时的**：容器删了/重启了，写在容器文件系统里的数据就没了。

要让数据**长期保存**（数据库、上传文件等），需要把外部存储「挂」进容器。Kubernetes 用一套抽象来管理：**PV / PVC / StorageClass**。

核心思路：**把「提供存储」和「使用存储」解耦**。

- 管理员提供存储资源（PV / StorageClass）
- 用户只管申请（PVC），不用关心底层是什么盘

---

## 二、三大核心对象：PV / PVC / StorageClass

| 对象 | 全称 | 作用域 | 通俗理解 |
|------|------|--------|----------|
| **PV** | PersistentVolume | 集群级 | 一块**已经存在**的存储（管理员准备好的「房子」） |
| **PVC** | PersistentVolumeClaim | 命名空间级 | 用户对存储的**申请**（「我要一间 2G 的房子」） |
| **StorageClass** | StorageClass | 集群级 | 描述存储类别、供应器和绑定策略；有可用供应器时可动态创建 PV |

关系：

```
静态：管理员建 PV  ←绑定→  用户建 PVC  ←挂载→  Pod
动态：用户建 PVC → StorageClass 指定的供应器造 PV → 绑定 → Pod
```

### 2.1 绑定条件（PVC 找 PV，三者必须同时满足）

1. PVC 请求的 `storage` **≤** PV 的 `capacity`（容量够）
2. `accessModes` **兼容**（访问模式匹配）
3. `storageClassName` **完全一致**（含都为空字符串的情况）

> **PVC 一直 Pending** 基本就是这三条里有一条不满足。

---

## 三、访问模式 Access Modes

描述存储能被怎样挂载。

| 模式 | 缩写 | 含义 |
|------|------|------|
| ReadWriteOnce | RWO | 被**单个节点**读写（最常用） |
| ReadOnlyMany | ROX | 被**多个节点**只读 |
| ReadWriteMany | RWX | 被**多个节点**读写（需要底层存储支持） |
| ReadWriteOncePod | RWOP | 被**单个 Pod**读写（v1.29+） |

> `hostPath` / local 本地盘不能提供跨节点共享数据的 RWX 语义。要跨节点 RWX，通常需要 NFS、分布式文件系统或支持该能力的 CSI 存储。

---

## 四、回收策略 Reclaim Policy

删除 PVC 后，PV 怎么处理。

| 策略 | 行为 |
|------|------|
| **Retain** | PV **保留**，数据不删，需管理员手动清理（安全，考试常用） |
| **Delete** | 删除 PV，并由支持该能力的供应器删除底层存储（动态云盘常见） |
| ~~Recycle~~ | 已废弃，不用管 |

在 PV 里配置：

**【YAML 片段｜不可单独 apply】**

```yaml
spec:
  persistentVolumeReclaimPolicy: Retain
```

## 五、Volume Mode

| 模式 | 说明 |
|------|------|
| **Filesystem** | 挂载为一个目录（默认，99% 的情况） |
| **Block** | 挂载为原始块设备（特殊场景） |

## 六、静态供应 vs 动态供应

### 6.1 静态供应：local PV 完整实验

管理员先在指定节点准备目录，再手动创建 PV；PVC 只负责匹配并绑定这个已经存在的 PV。本实验把数据固定放在 `k8s-worker1` 的 `/data/cka-local-pv`。

#### 第 1 步：确认节点名称与标签

`nodeAffinity` 必须使用集群实际的节点标签值，先在控制端确认：

**【控制端 WSL】**

```bash
kubectl get nodes -o wide
kubectl get node k8s-worker1 -o jsonpath='{.metadata.labels.kubernetes\.io/hostname}{"\n"}'
```

预期第二条命令输出 `k8s-worker1`。如果输出不同，后面 PV 的 `values` 必须改成实际值。

#### 第 2 步：在 worker1 准备底层目录

**【k8s-worker1】**

```bash
sudo mkdir -p /data/cka-local-pv
sudo chmod 0777 /data/cka-local-pv
printf 'created on k8s-worker1\n' | sudo tee /data/cka-local-pv/node-source.txt
ls -ld /data/cka-local-pv
```

> `0777` 仅用于学习环境，便于任意容器用户写入。生产环境应按 Pod 的 UID/GID、`fsGroup` 和最小权限原则设置目录属主与权限。

#### 第 3 步：保存并应用完整资源

下面一个文件包含 Namespace、StorageClass、PV、PVC 和 Pod。`local` 卷必须配合 PV 的 `nodeAffinity`，这样调度器才知道数据实际位于哪个节点。

**【控制端 WSL｜保存为 `cka-local-storage.yaml` 并 apply】**

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: storage-lab
---
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: cka-local
provisioner: kubernetes.io/no-provisioner
volumeBindingMode: WaitForFirstConsumer
reclaimPolicy: Retain
---
apiVersion: v1
kind: PersistentVolume
metadata:
  name: cka-local-pv
spec:
  capacity:
    storage: 1Gi
  volumeMode: Filesystem
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: cka-local
  local:
    path: /data/cka-local-pv
  nodeAffinity:
    required:
      nodeSelectorTerms:
        - matchExpressions:
            - key: kubernetes.io/hostname
              operator: In
              values:
                - k8s-worker1
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: cka-local-pvc
  namespace: storage-lab
spec:
  accessModes:
    - ReadWriteOnce
  volumeMode: Filesystem
  resources:
    requests:
      storage: 1Gi
  storageClassName: cka-local
---
apiVersion: v1
kind: Pod
metadata:
  name: local-storage-app
  namespace: storage-lab
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "sleep 36000"]
      volumeMounts:
        - name: data
          mountPath: /data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: cka-local-pvc
```

**【控制端 WSL】**

```bash
kubectl apply -f cka-local-storage.yaml
kubectl get sc cka-local
kubectl get pv cka-local-pv
kubectl get pvc,pod -n storage-lab -o wide
```

预期 PVC 和 PV 都是 `Bound`，Pod 为 `Running`，且 Pod 的 `NODE` 是 `k8s-worker1`。`WaitForFirstConsumer` 会等到 Pod 出现后，结合 Pod 调度条件与 PV 节点亲和性完成绑定。

| PV 字段 | 含义 |
|---------|------|
| `capacity.storage` | PV 声明可提供的容量 |
| `accessModes` | 支持的访问模式 |
| `persistentVolumeReclaimPolicy` | PVC 删除后的回收策略 |
| `storageClassName` | 类名，PVC 必须与其匹配 |
| `local.path` | 指定节点上的本地目录 |
| `nodeAffinity` | 把 PV 与实际存放数据的节点关联 |

### 6.2 `no-provisioner` 不等于动态供应

`provisioner: kubernetes.io/no-provisioner` 的准确含义是：**没有供应器负责创建或删除底层卷，也不会根据 PVC 自动创建 PV**。它常用于 local PV，但 local PV 仍需管理员像 6.1 那样预先准备目录并手动创建 PV，因此属于**静态供应**。

下面只是关键字段说明，不是可应用的完整 StorageClass：

**【YAML 片段｜不可单独 apply】**

```yaml
provisioner: kubernetes.io/no-provisioner
volumeBindingMode: WaitForFirstConsumer
```

真正的动态供应需要集群安装并配置可工作的存储供应器，例如云盘 CSI、NFS 外部供应器或其他 CSI 驱动。流程才是：

```text
PVC → StorageClass 指定的供应器/CSI 驱动 → 自动创建 PV 与底层卷 → 绑定
```

| `volumeBindingMode` | 说明 |
|---------------------|------|
| **Immediate** | PVC 创建后立即尝试供应或绑定；可能尚未考虑 Pod 最终所在节点 |
| **WaitForFirstConsumer** | 等使用 PVC 的 Pod 出现后再结合调度拓扑供应或绑定；local PV 通常使用它 |

查看集群中现有 StorageClass 及供应器：

**【控制端 WSL】**

```bash
kubectl get storageclass
kubectl get storageclass -o custom-columns='NAME:.metadata.name,PROVISIONER:.provisioner,BINDING:.volumeBindingMode'
```

> 名字后带 `(default)` 的是默认 StorageClass。PVC 省略 `storageClassName` 时通常使用默认类；显式写 `storageClassName: ""` 则表示不使用 StorageClass。

## 七、Pod 挂载 PVC

Pod 挂载 PVC 的关键字段如下；完整可运行资源见 6.1。

**【YAML 片段｜不可单独 apply】**

```yaml
volumeMounts:
  - name: data
    mountPath: /data
volumes:
  - name: data
    persistentVolumeClaim:
      claimName: cka-local-pvc
```

| 字段 | 含义 |
|------|------|
| `volumeMounts.name` | 引用 `volumes` 里定义的卷 |
| `volumeMounts.mountPath` | 挂到容器内哪个目录 |
| `volumes.persistentVolumeClaim.claimName` | 使用的 PVC 名 |

### 7.1 写入数据并交叉验证

先从 Pod 写入文件：

**【控制端 WSL】**

```bash
kubectl exec -n storage-lab local-storage-app -- sh -c \
  'echo "persistent-data-$(date +%s)" > /data/persist.txt'
kubectl exec -n storage-lab local-storage-app -- cat /data/persist.txt
kubectl exec -n storage-lab local-storage-app -- cat /data/node-source.txt
```

再直接从数据所在节点确认同一个文件：

**【k8s-worker1】**

```bash
sudo cat /data/cka-local-pv/persist.txt
sudo ls -la /data/cka-local-pv
```

### 7.2 删除并重建 Pod，验证持久化

删除 Pod 不会删除 PVC、PV 或 worker1 上的数据。因为原清单中仍有 Pod 定义，可只重新应用该文件让 Pod 重建：

**【控制端 WSL】**

```bash
kubectl delete pod local-storage-app -n storage-lab
kubectl apply -f cka-local-storage.yaml
kubectl wait --for=condition=Ready pod/local-storage-app -n storage-lab --timeout=120s
kubectl get pod local-storage-app -n storage-lab -o wide
kubectl exec -n storage-lab local-storage-app -- cat /data/persist.txt
```

最后一条命令仍能读到原内容，说明数据生命周期独立于 Pod；重建后的 Pod 仍会被调度到 `k8s-worker1`。

### 7.3 完整清理闭环

先删除使用卷的 Pod，再删除 PVC。此时可观察 `Retain` 使 PV 进入 `Released`，然后由管理员显式删除 PV，最后删除 StorageClass 和 Namespace：

**【控制端 WSL】**

```bash
kubectl delete pod local-storage-app -n storage-lab
kubectl delete pvc cka-local-pvc -n storage-lab
kubectl get pv cka-local-pv
kubectl delete pv cka-local-pv
kubectl delete storageclass cka-local
kubectl delete namespace storage-lab
kubectl get pv cka-local-pv
kubectl get namespace storage-lab
```

删除 PVC 后，`Retain` 会保留 PV 对象和底层数据供管理员处置；随后执行 `kubectl delete pv` 是管理员的显式决定。`Retain` 不是“无论如何都阻止管理员删除 PV”。

确认 Kubernetes 对象清理后，再到数据所在节点删除实验目录：

**【k8s-worker1】**

```bash
sudo test -f /data/cka-local-pv/persist.txt && sudo cat /data/cka-local-pv/persist.txt
sudo rm -rf /data/cka-local-pv
sudo test ! -e /data/cka-local-pv && echo "worker directory cleaned"
```

> 执行 `rm -rf` 前务必再次核对路径。节点目录不会被 `kubernetes.io/no-provisioner` 或 `Retain` 自动删除。

## 八、其他常用 Volume 类型

除了 PVC，Pod 还能直接挂这些卷：

| 类型 | 用途 | 生命周期 |
|------|------|----------|
| **emptyDir** | Pod 内容器间共享的临时目录 | 随 Pod 删除而消失 |
| **hostPath** | 挂节点上的目录 | 与节点绑定（考试练习常用） |
| **configMap** | 把 ConfigMap 变成文件 | 见第二部分 |
| **secret** | 把 Secret 变成文件 | 见第二部分 |

下面仅展示 Pod `spec.volumes` 中的字段：

**【YAML 片段｜不可单独 apply】**

```yaml
volumes:
  - name: cache
    emptyDir: {}                # 临时空目录
  - name: config-vol
    configMap:
      name: app-config
  - name: secret-vol
    secret:
      secretName: db-creds
```

### 8.1 hostPath / local PV 的节点局限与生产边界

- `hostPath` 直接引用 Pod 所在节点的路径，本身不提供 PV/PVC 抽象，也不会保证 Pod 重建后仍落到原节点；同一路径在不同节点上是不同数据。
- `local` PV 比 `hostPath` 更适合表达节点本地持久盘：它通过 PV/PVC 管理，并要求 `nodeAffinity` 描述存储所在节点，但数据仍然不能随 Pod 漂移到其他节点。
- `ReadWriteOnce` 表示卷可由一个节点读写，不等于“只能有一个 Pod”。同一节点上的多个 Pod 是否能同时访问，还取决于存储实现和应用并发安全。
- 节点宕机、磁盘损坏或节点被移除时，本地数据可能不可用；Kubernetes 不会自动把该目录复制到其他节点。
- `Retain` 只保留 PV/底层数据供管理员处置；`Delete` 是否真能删除底层存储依赖实际供应器。对 `no-provisioner` 的 local PV，不应期待 Kubernetes 自动删除节点目录。
- 多节点生产环境若需要跨节点故障转移、快照、扩容或 RWX，应使用具备相应能力的 CSI/网络/分布式存储，并验证 StorageClass、拓扑、备份和灾难恢复方案。单节点 `hostPath` 或手工 local PV 更适合考试练习、缓存或明确接受节点绑定的数据。

## 九、存储故障排查

**【控制端 WSL】**

```bash
kubectl get pvc -n storage-lab                         # STATUS 是否 Bound
kubectl describe pvc cka-local-pvc -n storage-lab     # Events 里看 Pending 原因
kubectl get pv                                         # Available/Bound/Released
kubectl describe pv cka-local-pv
kubectl describe pod local-storage-app -n storage-lab # 看调度、挂载事件
```

**PVC 一直 Pending 的排查顺序：**

1. `storageClassName` 是否和 PV / 存在的 SC 一致？
2. `accessModes` 是否兼容？
3. 申请容量是否 ≤ PV 容量？
4. `local` PV 的 `nodeAffinity` 是否匹配真实节点标签？
5. 使用 `WaitForFirstConsumer` 时，是否已经创建了消费该 PVC 的 Pod？
6. 是否根本没有匹配的 PV，且 StorageClass 的供应器并不能动态创建 PV？

| PV 状态 | 含义 |
|---------|------|
| `Available` | 空闲，未绑定 |
| `Bound` | 已绑定到某 PVC |
| `Released` | PVC 已删但 PV（Retain）还未回收 |

## 十一、易错点与自检清单

### 易错点

- **绑定三条件**：容量、accessModes、storageClassName，缺一不绑。
- `storageClassName: ""`（空串）只匹配**同样没有 SC** 的 PV，不等于「不写」。
- `kubernetes.io/no-provisioner` **不会**根据 PVC 动态创建 PV。
- local PV 必须配置正确的 `nodeAffinity`；数据不能自动跨节点迁移。
- 删除顺序：**先删 Pod → 再删 PVC → 最后处理 PV**。
- Retain 的 PV 在 PVC 删除后进入 `Released`，不会自动可用，需手动处理。

### 自检清单

- [ ] 能解释 PV / PVC / StorageClass 区别
- [ ] 能创建带 `nodeAffinity` 的 local PV + PVC 并绑定成功（Bound）
- [ ] 能解释 `no-provisioner` 为什么仍是静态供应
- [ ] 能区分 no-provisioner 与真正 CSI/外部供应器的动态供应
- [ ] 能让 Pod 挂载 PVC，并在重建 Pod 后验证数据仍存在
- [ ] 能排查 PVC Pending（绑定条件、拓扑和供应器）
- [ ] 理解回收策略 Retain / Delete
